Abstract

Modern neural network frameworks treat the network as an object and the programming language as a separate tool used to manipulate it. This separation is practical but fundamentally misaligned: neural networks are graph-structured, self-adjusting computational artifacts, and the languages used to build them are not. The result is a persistent impedance mismatch — networks that cannot be directly inspected as programs, architectures that cannot be composed with the same tools used to compose code, and training processes that are structurally isolated from the languages that orchestrate them.

This paper specifies NeuralLISP, a Common Lisp-family language in which neural networks are not a library, not a framework, and not an abstraction layer. They are functions. A network is defined with defnet the way a function is defined with defun — it takes arguments, returns values, and composes with other functions. The difference is that defnet produces a differentiable function: one whose parameters are tracked by the runtime and whose gradient can be computed with respect to any loss.

NeuralLISP defines seven primitive forms: defactivation, dense, conv, recur, attend, defnet, and with-training. Above them, defmodule provides reusable parameterized subnetworks, defagent extends naturally to reinforcement learning, and defsymbolic bridges neural and symbolic reasoning — encoding logical constraints, domain rules, and formal fallacies as differentiable guards on network behavior. Because NeuralLISP is homoiconic — code and data share the same representation — a running network can inspect and rewrite its own architecture using the same operations used to manipulate any other program. Self-modification is not a designed feature. It is an emergent consequence of treating networks as code in a language where code is data.[2]

Section 1

Introduction: The Impedance Mismatch Problem

Neural networks are functions. Not metaphorically — structurally. A feedforward network takes an input tensor, passes it through a sequence of parameterized transformations, and returns an output tensor. That is what a function does. Recurrent networks are more precisely stateful functions — functions from input and accumulated hidden state to output and new hidden state — but the functional character is preserved; recur forms handle this state threading internally, keeping the programmer-facing model uniform.

LISP is also, fundamentally, about functions. Not just in the sense that it supports functional programming — in the deeper sense that functions are the organizing principle of the language. Functions are values. They can be passed as arguments, returned from other functions, stored in variables, and composed into larger functions. The entire LISP standard library is built on the idea that the most powerful thing you can do with a computation is hand it to another computation as an argument.[2]

These two things — neural networks and LISP — are describing the same underlying structure from different directions. A neural network is a composed function built from parameterized transformations. LISP is a language designed to construct, manipulate, and evaluate composed functions. The alignment is not superficial. It goes all the way down to what both things fundamentally are.

And yet the tools we use today to build neural networks treat the language and the network as entirely separate concerns. You write Python. Separately, a framework constructs a computation graph behind the scenes. The network is an object your code talks to, not something your code is. You cannot pass a trained network to map. You cannot partially apply a layer. You cannot compose two networks with the same syntax you use to compose two functions, because networks and functions are different categories of thing in every framework that exists today.

NeuralLISP eliminates this distinction by refusing to make it in the first place. In NeuralLISP, a network is defined the way a function is defined:

(defnet classifier (x)
  (-> x
      (dense  256 relu  :drop 0.2)
      (dense  128 relu  :drop 0.2)
      (dense  64  relu)
      (output 10  softmax)))

classifier is a callable. You invoke it with (classifier input-tensor). You pass it to other functions. You compose it with other networks. The only difference between defnet and defun is that defnet produces a differentiable function — one whose parameters are tracked by the runtime and whose gradient flows back through every layer, through every composed activation, automatically.

The activation functions — relu, softmax — are not keywords dispatched on internally by the framework. They are symbols bound to functions in the environment, defined with defactivation, carrying both a forward computation and a backward gradient. They can be composed:

(dense 256 (compose batch-norm relu))
(dense 256 (partial leaky-relu 0.2))
(dense 256 (lambda (x) (* x (tanh (log (+ 1 (exp x)))))))  ; mish

This design has three consequences developed in full throughout this paper. First, networks compose with the same syntax used to compose anything else in LISP. Second, the space of expressible activations is unbounded by what the language designers anticipated — any differentiable function is valid. Third, because NeuralLISP is homoiconic, a running network can reach into its own structure and rewrite it. Self-modification is not an API. It is what happens when you take homoiconicity seriously and apply it to something that matters.

Section 2

Why LISP: The Homoiconic Foundation

In 1960, John McCarthy published a paper that introduced something unusual into the history of computing. He was not primarily trying to build a programming language. He was trying to find a notation for computation clean enough to reason about mathematically. What he arrived at, almost as a side effect, was a language in which programs and the data those programs operate on have the same structure — S-expressions. The property that falls out of this decision is called homoiconicity: code and data are the same kind of thing.[2]

McCarthy's paper established three things that remain the foundation of NeuralLISP's design. First, that a small number of primitive operations — car, cdr, cons, eq, atom, cond, and function application — are sufficient to build any computation expressible in the language. Second, that functions are data: a function can be represented as a list, stored, passed as an argument, and evaluated. Third, that the evaluator itself can be written in LISP, and is short enough to fit on a page. These three properties together mean that a LISP program can treat any other LISP program — including itself — as data to be read, modified, and re-evaluated.

This is not a technical curiosity. It is the most consequential design decision ever made in a programming language. Sheard and Peyton Jones's work on Template Haskell demonstrated that a type-safe macro system could be integrated into a compiled language, enabling compile-time program generation with strong correctness guarantees.[14] NeuralLISP's Phase 1 shape inference is analogous — static analysis made possible by the fact that network definitions are inspectable at read time. But Common Lisp's macro system, codified by Steele,[1] goes further: it operates on arbitrary list structure rather than a restricted grammar, giving NeuralLISP's architecture macros the ability to generate, validate, and transform network definitions as freely as any other list manipulation.

Lisp and the Origins of Artificial Intelligence

Lisp was conceived explicitly for artificial intelligence. At the 1956 Dartmouth conference where John McCarthy coined the term "artificial intelligence," McCarthy sought a notation suitable for symbolic computation and mathematical reasoning about programs. By 1960, Lisp emerged — a language in which programs and data share the same list structure (S-expressions). This homoiconicity allows a program to treat other programs (or itself) as data.

Early AI was symbolic. Lisp excelled because it made recursion, first-class functions, and self-modification natural. Landmark systems such as SHRDLU (natural language understanding in a blocks world) and MACSYMA (symbolic algebra) were built in Lisp dialects. In the 1980s, specialized Lisp Machines from Symbolics, LMI, and Xerox ran entire operating systems and applications written in Lisp — the operating system was written in the same language as the applications, and every component was inspectable and modifiable at runtime. Lisp was, for a time, the de facto language of AI research.

The rise of statistical machine learning later shifted focus to numeric computation on commodity hardware. Yet the underlying alignment never disappeared: neural networks are composed differentiable functions — precisely the kind of structure Lisp was designed to manipulate, compose, and reason about.

NeuralLISP proposes to revive this original alignment. Networks would not be opaque objects; they would be modifiable programs in the language itself. The self-modifying MNIST classifier described in Appendix A, the neural scheduler proposed for a bare-metal hypervisor, the neuro-symbolic guards of defsymbolic, and the DXF round-trip are all direct descendants of McCarthy's 1960 vision — a language that can think about itself, applied to a domain where thinking about yourself is the whole point.

What Homoiconicity Actually Means

When the programmer writes a defnet form, that form is a list. The first element is the symbol defnet. The network's topology — every layer, every activation, every keyword — is nested list structure. And every tool that operates on LISP lists operates on that structure. The macro expander can transform it before evaluation. The reader can validate it at parse time. The evaluator can re-evaluate a modified version at runtime. None of this requires special machinery. It comes for free from the decision to write networks in LISP.

Functions as the Organizing Principle

In Common Lisp, a function is a first-class value. You bind it with defun, store it with #', pass it to mapcar, return it from another function with lambda, and compose it using standard higher-order operations. NeuralLISP builds on this directly. A layer form like (dense 256 relu) evaluates to a differentiable function. A defnet form evaluates to a differentiable function. The -> threading macro composes these functions into a pipeline. The pipeline is a function. All the way up and all the way down, the organizing principle is the same.

The -> threading macro follows thread-first semantics — the value threads as the first argument of each subsequent form — which is the correct choice since layer forms take their input tensor as the first argument. This convention is established in Clojure and widely understood; NeuralLISP adopts it directly.[13] Both -> chains and regular function calls mix freely inside a defnet body, since the body is ordinary LISP.

The Macro System as Architecture Engine

LISP macros operate at read time, before evaluation. A macro receives a list and returns a different list — the evaluator never sees the macro, only its result. This is the correct mechanism for expressing neural architecture families for two reasons. First, a macro can generate arbitrarily large defnet forms from a compact declaration. Second, a macro can validate structure before the network is built — catching shape mismatches at read time, before a single weight is initialized. NeuralLISP distinguishes between macro keyword arguments, which configure the macro's generation behavior, and primitive positional arguments, which appear in the expanded forms.

Why Common Lisp Specifically

NeuralLISP is built in the tradition of Common Lisp and AutoLISP rather than Scheme or Clojure for practical reasons. Common Lisp has defmacro, a full macro system operating on arbitrary list structure. It has a condition system — a structured mechanism for signaling and handling exceptional situations more powerful than exception handling in any other mainstream language. stop-training is a condition, not a flag. shape-mismatch is a condition, not an exception. And SBCL compiles Common Lisp to native machine code with type declarations that achieve near-C performance on tight numerical loops — making a pure CL tensor backend viable without any foreign function interface. AutoLISP demonstrated that a Common Lisp-adjacent dialect embedded inside a complex domain-specific host could be used productively by domain experts who had no interest in becoming language theorists.[5] NeuralLISP applies the same philosophy to machine learning.

The Evaluator as a Learning Loop

The LISP read-eval-print loop takes an expression, evaluates it, returns a result, and waits for the next expression. Neural network training takes a batch, runs a forward pass, computes a loss, runs a backward pass, updates weights, and waits for the next batch. These are structurally the same loop. In NeuralLISP, with-training is the REPL extended with gradient semantics. The language that defines the network is the same language that trains it, modifies it, and reasons about it. There is one environment, one evaluator, one loop.

Section 3

Core Primitives: The Language Specification

NeuralLISP has seven primitive forms from which every neural network can be expressed: defactivation, dense, conv, recur, attend, defnet, and with-training. Everything else — every architecture macro, every utility, every standard library network — is built from these seven things using the macro system and the standard LISP toolkit.

The rule governing argument style is simple and consistent throughout: if a parameter is always present and carries the primary meaning of the form, it is positional. If it is optional, an override, or a behavioral annotation, it is a keyword. No primitive form has more than three positional arguments. Every keyword argument is genuinely optional with a sensible default.

Companion forms. pool, flatten, and norm are derived companion forms, not primitives. They appear frequently alongside conv and dense but do not constitute independent primitives — each is a thin wrapper over tensor operations expressible in terms of the existing forms. The output form is a required terminal form within defnet bodies, not a standalone primitive; it declares the network's output width and final activation. defweights enables explicit weight sharing across layers — binding named weight tensors for use by multiple layer forms, essential for siamese networks and shared embeddings — following the same pattern as defactivation. defloss defines custom loss functions on the same model as defactivation.
Tensors as first-class values. A tensor in NeuralLISP is a first-class type — an nl-tensor struct backed by a row-major (simple-array single-float (*)), with shape metadata, gradient storage, and autograd tape linkage. Standard arithmetic — tensor-add, tensor-sub, tensor-mul, tensor-div — operates element-wise on the flat data vector. Reduction functions — tensor-mean, tensor-sum, tensor-max — accept tensors and optional axis arguments. Structural operations — tensor-cat, tensor-reshape, tensor-transpose, tensor-flatten — are standard library functions. This is why the residual connection (+ x h) in defmodule, the ensemble (mean (mapcar ...)) in defnet, and the U-Net skip connections (cat e4 e3) in the diffusion macro are ordinary expressions rather than architectural primitives: tensors are values, arithmetic is defined on them, and ordinary function calls compose them. Tensor construction: (torch-randn shape), (torch-zeros shape), (torch-ones shape). Weight initialization: (xavier-uniform shape), (kaiming-uniform shape). No CFFI, no foreign pointers — the tensor is a Common Lisp struct, inspectable with describe, serializable with write, and garbage-collected normally.

3.1 defactivation — Differentiable Functions as Values

Before layer forms can be specified, activation functions must be. They are not keywords in NeuralLISP. They are values — callable objects carrying both a forward computation and a backward gradient, registered in the environment like any other symbol.

(defactivation relu
  :forward  (lambda (x) (max 0.0 x))
  :backward (lambda (x) (if (> x 0.0) 1.0 0.0)))

(defactivation gelu
  :forward  (lambda (x) (* x (phi x)))
  :backward (lambda (x) (+ (phi x) (* x (gauss x)))))

(defactivation leaky-relu
  :forward  (lambda (slope x) (if (> x 0) x (* slope x)))
  :backward (lambda (slope x) (if (> x 0) 1.0 slope)))

; Standard library ships with: relu, sigmoid, tanh, gelu,
; softmax, linear, leaky-relu (default slope 0.2),
; swish, mish, elu, selu — all as named symbols.

After defactivation, relu is a symbol in the environment bound to a callable. Because activations are functions, they compose with the tools LISP already provides:

; Composition — batch-norm then relu
(dense 256 (compose relu batch-norm))

; Partial application — leaky relu with fixed slope
(dense 256 (partial leaky-relu 0.15))

; Lambda inline — mish activation, no defactivation required
(dense 256 (lambda (x) (* x (tanh (log (+ 1.0 (exp x)))))))

; Shared activation bound to a variable — changing the variable
; changes all layers that reference it simultaneously
(defvar my-act (partial leaky-relu 0.15))
(defnet model (x)
  (-> x
      (dense 256 my-act)
      (dense 128 my-act)   ; same function object, not a copy
      (output 10  softmax)))

autodiff — the complement to defactivation. defactivation requires explicit :forward and :backward definitions — the programmer specifies both the function and its derivative. autodiff is the escape hatch when you want the gradient computed automatically from an arbitrary tensor expression rather than specified by hand. It wraps any lambda in NeuralLISP's native tape-based reverse-mode AD, recording every operation to build the backward pass automatically:

; autodiff traces tensor operations and computes gradients automatically
(defvar l2-norm-act
  (autodiff (lambda (x)
              (/ x (+ (torch-norm x 2) 1e-8)))))

(dense 256 l2-norm-act)   ; gradient flows through — no :backward needed

; Any differentiable tensor expression qualifies
(defvar scaled-sigmoid
  (autodiff (lambda (x) (* 1.7159 (tanh (* 0.6667 x))))))

The constraint is that every operation inside the lambda must use NeuralLISP's tensor-aware arithmetic — the ad-add, ad-mul, ad-tanh, and so on that record to the autograd tape — rather than raw Common Lisp arithmetic on scalar numbers. Within that constraint, autodiff makes the entire tensor standard library differentiable: any expression you can write with tensor arithmetic, you can use as an activation function, a loss function, or an intermediate computation inside a defnet body. defactivation is the right choice when you have a clean mathematical definition and want the gradient explicitly stated. autodiff is the right choice when you want to experiment freely and let the tape work out the derivative.

A dense form evaluates to a differentiable function from input tensor to output tensor — a closure over a weight matrix W and bias b, applying activation(xWᵀ + b). Weight and bias tensors are registered with the autograd system at evaluation time.

The :as keyword and connect. The :as keyword gives a layer a local name within a defnet body, making it referenceable by inspect and rewrite by name rather than by index. For non-sequential topologies — branches, merges, and skip connections — the natural NeuralLISP idiom is let* with tensor arithmetic, since tensors are values and + is defined on them:

; Skip connection — let* names intermediate tensors, + adds them
(defnet residual-block (x)
  (let* ((h (-> x
               (dense 256 relu :as 'hidden)
               (dense 256 relu))))
    (+ x h)))   ; x bypasses both layers — residual connection

; Branch and merge — two parallel paths, concatenated
(defnet two-path (x)
  (let* ((path-a (-> x (dense 128 relu)))
         (path-b (-> x (dense 128 gelu)))
         (merged (cat path-a path-b)))  ; concatenate along feature dim
    (-> merged (output 10 softmax))))

; Named layers are accessible to inspect and rewrite by :as name
; rather than by index — useful when topology is complex
(inspect residual-block :as 'hidden)
; => (dense 256 relu :as hidden)
(rewrite residual-block :as 'hidden (dense 256 gelu :as 'hidden))

For the majority of non-sequential patterns — residuals, U-Net skips, parallel branches — let* and tensor arithmetic express the topology more clearly than an explicit wiring form. The connect utility form (not a primitive) is available for programmatically wiring graphs too large to express legibly as let* chains, but in practice the paper's architecture macros use let* throughout.

  (-> x

      (dense 128 relu :weights shared-w2 :init :frozen)

3.4 recur — Recurrent Layer

Cell type — lstm, gru, rnn — is positional because it fundamentally determines the layer's computational structure, parameter count, and gradient behavior.[7] These cell type symbols are values in the environment, not keywords — they can be stored in variables and selected programmatically.

(recur width cell)
(recur width cell :bidir t :drop rate)

; Cell type as a runtime value
(defvar my-cell (if use-lstm 'lstm 'gru))
(recur 256 my-cell)

; Stacked bidirectional LSTM
(defnet sequence-model (x)
  (-> x
      (recur 256 lstm :bidir t :drop 0.3)
      (recur 256 lstm :bidir t :drop 0.3)
      (output 10 softmax)))

3.6 defnet — A Differentiable Function

defnet is specified deliberately to resemble defun as closely as possible. The body is almost always a -> threading form describing the forward computation. -> follows thread-first semantics: (-> x f g h) expands to (h (g (f x))). Its definition is four lines of standard Common Lisp:

(defmacro -> (val &rest forms)
  "Thread val as the first argument through each form left-to-right.
   (-> x f g h) expands to (h (g (f x)))"
  (if (null forms) val
      `(-> ,(if (listp (car forms))
                 `(,(caar forms) ,val ,@(cdar forms))
                 `(,(car forms) ,val))
            ,@(cdr forms))))

defnet additionally supports multiple return values via values, inherited from Common Lisp. A variational autoencoder encoder returning both mean and log-variance is a natural application — no special multi-output primitive required:

; Single input — sequential
(defnet classifier (x)
  (-> x
      (dense  256 relu :drop 0.2)
      (dense  128 relu :drop 0.2)
      (output 10  softmax)))

; Multiple return values — VAE encoder returns mean and log-variance.
; The two projection layers must persist across calls — define them
; at network definition time by closing over named layer functions.
(defnet vae-encoder (x)
  (let* ((h       (-> x (dense 256 relu) (dense 128 relu)))
         (mu      (-> h (dense 32 linear :as 'mu-proj)))
         (log-var (-> h (dense 32 linear :as 'logvar-proj))))
    (values mu log-var)))

; Called with multiple-value-bind — standard Common Lisp
(multiple-value-bind (mu log-var)
    (vae-encoder input-batch)
  (reparameterize mu log-var))

; Networks compose like functions — because they are functions
(defnet ensemble (x)
  (mean (mapcar (lambda (m) (funcall m x))
                 (list model-a model-b model-c))))

Unit testing networks follows directly from their function nature — any test framework that works on Common Lisp functions works on defnet forms:

; Shape contract test
(assert (equal (tensor-shape (encoder (torch-randn '(1 784))))
               '(1 32)))

; Module preserves spatial shape through residual
(assert (equal (tensor-shape (residual-block (torch-randn '(1 64 64)) 64 64))

Topology can be built programmatically before threading — the macro expander sees a fully formed expression, the reader validates shapes, and quasiquote splices the generated layers:

(defvar enc-layers
  (loop for w in '(512 256 128)
        collect `(dense ,w relu :drop 0.2)))

(defnet encoder (x)
  (-> x
      ,@enc-layers
      (output 64 linear)))

3.7 with-training — The Training Macro

with-training must be a macro because its body — code evaluated each epoch — cannot be eagerly evaluated at call time. stop-training is a Common Lisp condition, not a flag. Loss functions are defined with defloss following the same pattern as defactivation. The torch-* prefix in tensor constructors is a naming convention retained for familiarity; in the pure CL implementation these are native struct allocations. Training utilities such as make-data-loader, accumulate-loss, and epoch-mean-loss are NeuralLISP standard library functions.

Checkpointing saves both the network's definition list and its weight tensors. The definition is written as readable LISP source; the weights are a binary tensor file alongside it. Loading restores both: the definition is eval-ed to reconstruct the network symbol, and the weights are loaded into its parameter tensors. This means a checkpoint is not just a weight snapshot — it is a complete, self-describing program artifact.

; Checkpoint saves definition + weights
(checkpoint classifier "/models/run-42")
; Writes: /models/run-42.lisp  — the defnet form as source
;         /models/run-42.nlt   — the weight tensors (native CL format)

; Load restores the live network
(load-checkpoint "/models/run-42")
; => evaluates run-42.lisp, loads weights, rebinds CLASSIFIER

The condition system handles the full range of training failures. stop-training is the clean exit. nan-loss and exploding-gradient are conditions that the training body can handle specifically — rolling back to a checkpoint, adjusting the learning rate, or logging a diagnostic — before deciding whether to continue or abort:

(with-training (net    classifier)
               (data   train-set :batch 64 :shuffle t)
               (loss   cross-entropy)
               (opt    (adam 0.001 :decay 1e-4))
               (epochs 100)
  (when (zero-mod epoch 10)
    (checkpoint classifier (format nil "/models/epoch-~d" epoch)))
  (when (< current-loss 0.01)
    (signal 'stop-training))
  ; Handle NaN loss — roll back to last checkpoint and halve lr
  (handler-case
      (when (nan-p current-loss)
        (load-checkpoint (format nil "/models/epoch-~d"
                                   (- epoch 10)))
        (halve-learning-rate opt)
        (format t "~&NaN at epoch ~d — rolled back, lr halved~%" epoch))
    (nan-loss (c)
      (format t "~&Unrecoverable NaN: ~a~%" c)
      (signal 'stop-training))))

(adam 0.001)
(sgd  0.01 :momentum 0.9 :nesterov t)
(schedule (adam 0.001) :warmup 1000 :decay cosine)

; Custom loss — defloss follows defactivation pattern

    (let ((ce (cross-entropy preds targets)))

Section 4

Macros as the Neural Architecture Engine

The seven primitives are the assembly language of NeuralLISP. The macro system is the compiler. An architecture macro takes a compact declaration of intent and expands it into a fully specified collection of defnet forms at read time. The evaluator never sees the macro — only its result.

In every framework that exists today, high-level architecture abstractions are classes defined by framework authors and sealed by API boundaries. To customize them you subclass and override. In NeuralLISP, the high-level abstractions are macros. Their expansions are readable. macroexpand gives you the full expanded form and the expansion is just more NeuralLISP. The programmer who uses defnet-transformer has exactly the same power as the programmer who wrote it.

4.1 How Architecture Macros Work

(defmacro dense-tower (name sizes activation output-width)
  `(defnet ,name (x)
     (-> x
         ,@(mapcar (lambda (w) `(dense ,w ,activation))
                    sizes)
         (output ,output-width softmax))))

; Used:
(dense-tower classifier (256 128 64) relu 10)

; Expands to:
(defnet classifier (x)
  (-> x
      (dense 256 relu)
      (dense 128 relu)
      (dense 64  relu)
      (output 10 softmax)))

The generation machinery is mapcar over a list of widths — the same mapcar that works on any list. Activation functions are passed through as symbols. The macro receives relu as a value and splices it into the expansion. No keyword dispatch, no special cases.

The adversarial combinator is the one place in NeuralLISP where the standard defnet / with-training model is insufficient. Most architectures are a single forward-pass function trained against a single loss. A GAN is two networks trained in opposition — the discriminator's loss depends on the generator's output, and the generator's loss depends on the discriminator's judgment. adversarial is a special form that registers this coupling:

; adversarial registers a minimax training pair
; Produced by defnet-gan macro — can also be written directly
(defnet image-gan (z x)
  (adversarial image-gan-generator
               image-gan-discriminator))

; What adversarial does internally:
; (adversarial gen disc) attaches three properties to the network symbol:
;   :gen-network  → image-gan-generator
;   :disc-network → image-gan-discriminator
;   :training-mode → :adversarial
; with-training reads :training-mode and alternates gradient updates:
;   Step 1 — freeze gen, update disc on real + generated batches
;   Step 2 — freeze disc, update gen against disc's judgment
; Both networks remain independently callable after training.

adversarial is not a macro that expands into primitive forms — it is a special registration form handled by the evaluator. It is the acknowledged exception to the paper's claim that all architecture patterns are expressible through function composition. The exception is honest: a minimax training relationship is not a function. It is a game, and games have a different structure than pipelines. adversarial is the narrowest possible special case — it registers two names and a training mode, and the rest is handled by with-training's existing machinery.

4.3 defmodule — Reusable Parameterized Subnetworks

The architecture macros in Section 4.2 expand to defnet forms at read time. Every expansion produces a new named network. This is correct for most purposes — a transformer macro that generates six uniquely named encoder blocks is exactly what we want when building a specific model.

But there is a different need that macros do not address cleanly: a reusable architectural pattern with explicit input and output contracts that can be instantiated multiple times with different dimensions inside a single defnet body. A residual block, an inception module, a squeeze-and-excitation unit — these are not one-off named networks. They are parameterized structural patterns that should behave like layer forms: callable with positional arguments, composable in -> chains, instantiating fresh weight tensors each time.

defmodule addresses this. It defines a reusable subnetwork with declared input arguments, an explicit body, and a structural contract validated at instantiation time. Unlike defnet, its dimension parameters are resolved when it is called, not when it is defined — making a single defmodule definition serve any compatible shape.

; Define a reusable residual block
(defmodule residual-block (x in-dim out-dim)
  (let ((h (-> x
               (norm  in-dim)
               (dense out-dim relu :drop 0.1)
               (norm  out-dim))))
    (+ x h)))   ; residual: tensor addition — not a special primitive

; Instantiate multiple times inside a defnet with different dims
(defnet resnet-6 (x)
  (-> x
      (conv   64  relu (3 3))
      (residual-block x 64  64)
      (residual-block x 64  64)
      (residual-block x 64  128)
      (pool   (2 2) avg)
      (flatten)
      (output 10 softmax)))

Each instantiation allocates a fresh set of weight tensors. Weight sharing across instantiations is achieved by passing a defweights symbol, the same mechanism used for layer-level sharing. The input contract — that (tensor-width x) equals in-dim — is validated at instantiation time; a module-contract-violation condition is signaled before any computation if dimensions are incompatible.

defnetdefmodule

With defmodule, the transformer encoder block — currently generated as six separate defnet forms by defnet-transformer — becomes a single defmodule instantiated six times. The macro expansion is more compact, the shared structural definition is visible in one place, and modifying the block definition propagates to all instantiations automatically on the next macroexpand.

Implementation. defmodule is a macro that expands to a defun augmented with a with-fresh-parameters call — a macro that allocates a new set of weight tensors at each call site, registers them with the autograd system, and closes over them in the returned function. The dimension parameters (in-dim, out-dim) are ordinary function arguments; the weight tensors are allocated using those dimensions when the function is called. This makes defmodule implementable entirely in terms of Phase 3 infrastructure with no new evaluator machinery.

4.4 Reinforcement Learning — defagent

The architecture macros in Sections 4.2 through 4.3 all define supervised or generative learning systems. Reinforcement learning is structurally different: the agent learns by interacting with an environment, receiving rewards, and updating its policy. But in NeuralLISP, RL is not structurally different at all. It is two defnet forms, a training loop, and a reward function — which is a defloss — and the existing machinery handles all of it. defagent makes this explicit.

(defagent rl-agent
  :policy  (defnet policy-net (state)
             (-> state
                 (dense 256 relu)
                 (dense 128 relu)
                 (output n-actions softmax)))
  :value   (defnet value-net (state)
             (-> state
                 (dense 256 relu)
                 (dense 128 relu)
                 (output 1 linear)))
  :env     cart-pole
  :gamma   0.99
  :reward  (lambda (state action next-state done)
             (if done -1.0 1.0)))

defagent expands to the two named defnet forms declared inline, a replay buffer bound in the environment, and an extended with-training invocation with :mode :actor-critic. The replay buffer is a circular vector with a fill pointer — a standard Common Lisp data structure providing O(1) push and O(1) random-access sampling, suitable for the random experience sampling that actor-critic training requires. Episodes are added with a ring-buffer push; batches are sampled by generating random indices and collecting the corresponding transitions. The actor-critic loss is a defloss computing policy gradient weighted by advantage estimates from the value network.

The reward function is a lambda — but it can be any callable, including a defnet. A learned reward model is just another network passed to :reward:

; Learned reward model — inverse RL in one clause
(defagent irl-agent
  :policy  (defnet policy-net  (state) ...)
  :value   (defnet value-net   (state) ...)
  :reward  (defnet reward-net  (state action) ...)  ; learned
  :env     maze-env)

; Self-modifying policy — grows when performance stagnates
(with-training (net   rl-agent)
               (data  (env-rollout cart-pole policy-net) :batch 64)
               (loss  (ppo-loss :clip 0.2))
               (opt   (adam 0.0003))
               (epochs 10000)
  (when (and (zero-mod epoch 500)
             (< (mean-reward-last 100) reward-threshold))
    (rewrite policy-net :insert-before :output
      (dense (* 2 (width-of policy-net :last-hidden)) relu)))
  (log-mean-reward epoch))

defagent is the clearest demonstration that NeuralLISP's design generalizes beyond supervised learning without requiring new primitives. RL is not a special case the language was extended to handle. It is a natural consequence of having differentiable functions as values, training loops as macro bodies, and loss functions as callables — and of the fact that defnet and lambda are the same kind of thing.

4.5 Macros Are Readable

Every macro can be fully expanded at the REPL with macroexpand. The expansion is readable NeuralLISP — every layer, every connection, every subnetwork — and the programmer can copy, modify, and use it directly. A macro package convention allows community-published architectures as ordinary Common Lisp packages: (require :neurallisp-mamba) makes defnet-mamba available. Packages are distributed as ASDF system definitions and installed via Quicklisp — the same mechanism used for all Common Lisp library distribution — making discovery, installation, and version pinning identical to any other dependency in the ecosystem. Domain-specific packages — :neurallisp-vision, :neurallisp-nlp, :neurallisp-rl — keep the core language small while extending its reach. Because user-defined macros expand to the same primitives as standard library macros, there is no distinction between first-party and third-party architectures at the language level.

Section 5

Self-Modification as an Emergent Property

Nothing in NeuralLISP was designed specifically to enable self-modification. There is no self-modification API, no special runtime mode, no flag to enable it. It emerges from two decisions made for completely independent reasons: networks are functions, and functions are values. Those two decisions have a consequence that was never designed and never required special machinery to achieve.

When a network is a function and a function is a value, a running network can reach into its own definition, read it, and rewrite it — through the same operations that reach into and rewrite any other value in a LISP program.

5.1 What a Network Definition Is

When the programmer writes a defnet form, defnet binds the symbol in the environment to a differentiable function and also stores the network's definition — the list structure of the form — as a property of that symbol. The definition and the function are two faces of the same object. inspect retrieves the definition as data. Because defnet stores its definition as a list property, copy-network is effectively (copy-list (get netname :definition)) combined with a snapshot of the current weight tensors, and restore-network is eval on that stored definition with weights restored.

In a conventional deep learning framework, a trained model is a fixed artifact whose architecture is determined before training begins and does not change — weight values update via gradient descent, but the graph those weights inhabit is static.[8] This is not a technical limitation. It is a consequence of treating the network as an object rather than a program. Objects have state. They do not rewrite their own class definitions at runtime. In NeuralLISP, there is no such constraint.

(inspect classifier)
; => (defnet classifier (x)
;      (-> x
;          (dense  256 relu  :drop 0.2)
;          (dense  128 relu  :drop 0.2)
;          (output 10  softmax)))

(inspect classifier :layer 2)
; => (dense 128 relu :drop 0.2)

(inspect classifier :layer 2 :pos 2)
; => relu     ; the activation — a symbol bound to a function

5.2 Rewriting a Live Network

Every rewrite validates shapes before modifying the network, preserves weights in unmodified layers, rebuilds the gradient graph lazily on the next forward pass, and appends to the modification history. The path argument :pos N addresses the Nth positional argument of a layer form.

; Replace a layer entirely
(rewrite classifier :layer 2
  (dense 256 relu :drop 0.3))

; Change only the activation function
(rewrite classifier :layer 2 :pos 2 'gelu)

; Insert before a position
(rewrite classifier :insert-before :layer 3
  (norm 128))

; Shared activation — change variable, all layers update
(defvar shared-act relu)
(defnet shared-model (x)
  (-> x
      (dense 256 shared-act)
      (dense 128 shared-act)
      (output 10 softmax)))
(setf shared-act 'gelu)   ; both layers now use gelu

; Modification history — a list, queryable with standard list ops
(history classifier)
; => ((t=0      :defined)
;     (epoch=12 :rewrite :layer 2 :from (dense 128 relu :drop 0.2)
;                                  :to   (dense 256 relu :drop 0.3))
;     ...)

5.3 Activation Functions as Modification Targets

Because activation functions are first-class values, they are the most surgically precise modification target. Changing a layer's activation preserves its weights, its width, and its position — only the transformation changes. This is the smallest possible architectural modification, and it is directly accessible as a symbol swap.

; Targeted replacement
(rewrite classifier :layer 2 :pos 2 'gelu)

; Universal replacement
(rewrite-all classifier :pos 2 'gelu)

; Predicate replacement — only layers currently using relu
(rewrite-where classifier
  (lambda (layer) (eq (layer-activation layer) 'relu))
  :pos 2 'gelu)

5.4 Self-Modification Inside the Training Loop

The with-training body is the natural site for self-modification. Any LISP expression valid at the REPL is valid here. Introspection utilities such as layer-gradient-norms, dying-layers, loss-stagnant-p, min-gradient-layer, and activation-successor are provided by the NeuralLISP standard library.

; Progressive depth growth
(with-training (net    growing-net)
               (data   train-set :batch 32)
               (loss   cross-entropy)
               (opt    (adam 0.001))
               (epochs 500)
  (when (and (zero-mod epoch 50)
             (> current-loss 0.4))
    (rewrite growing-net :insert-before :output
      (dense (* 2 (width-of growing-net :last-hidden))
             relu :drop 0.2)))
  (log-loss epoch))
; Activation mutation under vanishing gradients
(with-training (net    deep-net)
               (data   train-set :batch 64)
               (loss   cross-entropy)
               (opt    (adam 0.001))
               (epochs 300)
  (when (loss-stagnant-p :window 20 :delta 1e-4)
    (let* ((worst-idx    (min-gradient-layer deep-net))
           (current-act  (layer-activation deep-net worst-idx))
           (next-act     (activation-successor current-act)))
      (rewrite deep-net :layer worst-idx :pos 2 next-act)))
  (log-loss epoch))

5.5 Networks That Write Networks

Because network definitions are lists, a function that returns a list can return a defnet form, and eval evaluates any form in the current environment. A controller network whose output encodes architecture decisions can generate, build, and evaluate new networks. eval is the bridge — and it has been in LISP since 1960.[2]

(defun embedding->defnet (embedding name)
  (let* ((widths  (extract-widths embedding))
         ; extract-activations returns a list of activation symbols
         ; e.g. '(relu gelu relu softmax) — values, not keywords
         (acts    (extract-activations embedding))
         (layers  (mapcar (lambda (w a) `(dense ,w ,a))
                           widths acts)))
    `(defnet ,name (x)
       (-> x ,@layers (output 10 softmax)))))

; eval is the bridge — one function call
(eval (embedding->defnet controller-output 'generated-net))

The implications of this pattern extend further than the code suggests. The search space the controller is navigating is not an abstract latent space of architecture embeddings — it is the NeuralLISP grammar itself. Every valid output from the controller corresponds to a valid defnet form. Every form the controller learns to prefer is a form that trains well on the target task. The controller is not learning a representation of architectures — it is learning to write programs.

This connects directly to defagent from Section 4.4. The controller network and its evaluation loop are exactly an RL agent: the policy produces architecture descriptions, the environment evaluates them by training and measuring validation accuracy, and the reward signal trains the controller to produce better descriptions. Written in NeuralLISP, this is defagent where the action space is the set of valid defnet forms and the environment is the NeuralLISP evaluator itself:

(defagent architecture-agent
  :policy  (defnet arch-controller (task-embedding)
             (-> task-embedding
                 (dense 256 relu)
                 (dense 128 relu)
                 (output 32 linear)))  ; architecture embedding
  :value   (defnet arch-critic (task-embedding)
             (-> task-embedding
                 (dense 256 relu)
                 (output 1 linear)))
  :env     neurallisp-eval-env     ; the evaluator is the environment
  :reward  (lambda (embedding action result done)
             (validation-accuracy result)))  ; accuracy is reward

; The modification history records every architecture tried.
; (history architecture-agent) is a log of every defnet ever generated,
; its validation accuracy, and whether the agent kept or discarded it.
; Neural architecture search is now a first-class NeuralLISP program.

The modification history — already specified as a list attached to the network symbol — serves a second purpose here. Every architecture the controller generates, evaluates, and either keeps or discards is appended to the history with its reward signal. After training, (history architecture-agent) is a complete record of the search — every form tried, every accuracy measured, every decision made. The search is not just productive; it is fully auditable, replayable, and resumable across sessions via the checkpoint mechanism.

5.6 Constraints and Stability

NeuralLISP imposes four constraints making self-modification stable enough to be productive. Each is enforced by the runtime rather than relied on from the programmer.

Shape validation before commitment. Every rewrite validates the modified network's shape contract before touching the computation graph. If the modification produces an incompatible shape anywhere downstream, a shape-mismatch condition is signaled. The network is unchanged. The programmer can handle the condition or let it propagate:

; Attempting an incompatible rewrite signals a condition
(rewrite classifier :layer 2
  (dense 512 relu))  ; layer 3 expects 128-wide input, not 512

; => shape-mismatch condition:
;    Layer 3 (dense 64 relu) expects input width 128
;    Layer 2 rewrite produces output width 512
;    Network unchanged.

; Handle it — adjust the downstream layer too
(handler-case
    (rewrite classifier :layer 2 (dense 512 relu))
  (shape-mismatch (c)
    (rewrite classifier :layer 2 (dense 512 relu))
    (rewrite classifier :layer 3 (dense 64  relu))
    (format t "~&Shape fixed: ~a~%" c)))

Weight preservation by default. Layers not touched by a rewrite keep their weights. Only the replaced layer is reinitialized. The programmer can override with :preserve nil for a full reinit, or provide explicit weights to preserve through a structural change. The REPL's preview-rewrite shows exactly what will be preserved and what will be reinitialized before the modification is committed:

(preview-rewrite classifier :layer 2
  (dense 256 gelu :drop 0.3))
; =>
;   BEFORE: (dense 128 relu :drop 0.20)   params=32,896
;   AFTER:  (dense 256 gelu :drop 0.30)   params=65,792
;   shape contract: valid
;   layer 2: reinitialized (:xavier)
;   layers 1, 3, 4: weights preserved

Lazy gradient graph reconstruction. The computation graph rebuilds on the next forward pass after a rewrite, not immediately. Multiple rewrite calls in a single epoch body are batched into one rebuild, keeping the cost proportional to the number of batches rather than the number of modifications.

Modification history as a first-class auditable value. Every modification is appended to a list attached to the network symbol. replay-history rebuilds the network from its original definition by applying each recorded modification in sequence — useful for auditing a runaway self-modifier or reproducing a training run:

(history mnist-net)
; => ((t=0      :defined)
;     (epoch=50 :rewrite :layer 1 :from (dense 64 relu)
;                                  :to   (dense 64 gelu)
;                         :reason :dying-gradient)
;     (epoch=50 :insert  :before :output
;                         :form   (dense 128 relu :drop 0.15))
;     (epoch=75 :rewrite :layer 1 :key :drop 0.12)
;     ...)

; Replay to epoch 50 — rebuild network as it was at that point
(replay-history mnist-net :to-epoch 50)

; Full replay from scratch — useful for reproducibility audits
(replay-history mnist-net)

5.7 Safety, Stability, and the Rewrite Sandbox

The constraints of Section 5.6 are sufficient for research. For deployment — especially in an OS kernel context — they are not. A network rewriting itself in production must operate under the same discipline as a database transaction: propose, validate, commit or rollback. Revision 5 introduces the rewrite sandbox: a controlled environment in which self-modifications are proposed, validated, and optionally approved before they touch the live network.

with-rewrite-sandbox — propose, validate, approve. The sandbox wraps a block of rewrite calls in a transactional envelope: modifications are applied to a copy of the network, validated against a test suite or metric, and committed only if the validation passes. If any step fails, the network reverts atomically.

(with-rewrite-sandbox (net classifier)
  ;; All rewrites inside this body operate on a shadow copy.
  ;; The real network is untouched until commit.
  (rewrite classifier :layer 2 :pos 2 'gelu)
  (rewrite classifier :insert-before :output
    (dense 256 relu :drop 0.1))

  ;; Validate: run a test batch and check accuracy doesn't degrade
  (:validate
    (let ((acc (evaluate classifier *val-set* :metric :accuracy)))
      (> acc (* 0.95 *baseline-accuracy*))))

  ;; Optional: require human approval for production systems
  (:approve :human)     ; or :auto for autonomous operation
  (:rate-limit 3 :per :epoch))  ; max 3 rewrites per epoch

Rate limiting. A network that rewrites itself every batch is not learning — it is thrashing. :rate-limit enforces a maximum number of structural modifications per time window (epoch, episode, or wall-clock interval). This is the simplest safeguard against runaway self-modification and the most important one for any kernel-level deployment.

Approval hooks. For deployment in safety-critical systems, :approve :human would pause execution and present the proposed modification to an operator before committing. For research, :approve :auto would commit immediately after validation. The approval mechanism is itself extensible — :approve accepts any callable, enabling custom approval logic including multi-party sign-off or confidence thresholds.

5.8 Architectural Provenance: The Flight Recorder

In aerospace, every commercial aircraft carries a flight data recorder — an indestructible log of every parameter change, every control input, every anomaly. When something goes wrong, the recorder provides the complete causal chain that led to the failure. No guessing, no reconstruction, no reliance on memory.

NeuralLISP's modification history is the flight recorder for neural architectures. Every rewrite — its timestamp, the before-state, the after-state, the triggering condition, and the epoch context — would be appended to an immutable log. In the OS kernel context described in Section 10, this becomes critical: when a self-modifying scheduler makes a poor decision, the provenance log provides complete accountability. What changed, when, why, and what the network looked like before.

(history classifier)
;; => ((t=0      :defined)
;;     (epoch=12 :rewrite :layer 2 :from (dense 128 relu :drop 0.2)
;;                                  :to   (dense 256 relu :drop 0.3)
;;                                  :reason :loss-stagnant
;;                                  :approved-by :auto)
;;     (epoch=50 :sandbox-rejected :proposed (dense 512 gelu)
;;                                  :reason :accuracy-degraded)
;;     ...)

;; Replay to any point in time
(replay-history classifier :to-epoch 12)

;; Diff between any two points
(history-diff classifier :from-epoch 0 :to-epoch 50)

This addresses what would otherwise be the fundamental objection to self-modifying systems: the determinism paradox. How do you debug a system that changes its own structure? Answer: you record every change as first-class data, and the debugging tools operate on the history with the same power they operate on the network itself — because both are lists.

Section 6

Neuro-Symbolic Reasoning: defsymbolic

The neural-symbolic research tradition has long asked how statistical learning and logical reasoning can be combined. Most approaches treat the boundary between the two as a hard interface — a carefully designed bridge between fundamentally different kinds of computation. NeuralLISP takes a different position: in a homoiconic language where networks are programs and programs are data, the boundary between symbolic reasoning and neural computation is not a bridge to be built. It is a distinction that dissolves.

defsymbolic is the form that makes this dissolution explicit. It defines logical constraints, domain rules, and formal reasoning patterns as composable guards on network behavior. These guards operate in two modes: soft rules that add penalty terms to the loss function during training, and hard rules that constrain the rewrite system, preventing structural modifications that would violate specified invariants.

6.1 Soft Rules: Loss Penalties from Logic

A soft symbolic rule is a logical constraint that the network should satisfy but is not required to. During training, violations of soft rules would be added to the loss as differentiable penalty terms. The network would learn to satisfy the constraint through gradient descent — the symbolic rule becomes a shaping signal for the numeric optimization.

;; Define a symbolic constraint: output probabilities must sum to 1
(defsymbolic probability-constraint :soft
  :predicate (lambda (output)
               (approx-equal (tensor-sum output) 1.0 :tolerance 0.01))
  :penalty   (lambda (output)
               (tensor-square (- (tensor-sum output) 1.0)))
  :weight    0.1)

;; Domain-specific: medical diagnosis must not contradict known exclusions
(defsymbolic mutual-exclusion :soft
  :predicate (lambda (output)
               (;; conditions A and B cannot both be > 0.5
                (not (and (> (tensor-ref output 3) 0.5)
                          (> (tensor-ref output 7) 0.5)))))
  :penalty   (lambda (output)
               (* (tensor-ref output 3) (tensor-ref output 7)))
  :weight    1.0)

;; Attach constraints to a network
(attach-symbolic classifier
  '(probability-constraint mutual-exclusion))

;; During training, the effective loss becomes:
;;   L_total = L_task + sum(weight_i * penalty_i(output))

The penalty functions would be differentiable — they would participate in backpropagation through the autograd tape. The network would not merely check the constraint after the fact; it would learn to satisfy it through the same gradient mechanism it uses to minimize task loss. The symbolic constraint and the numeric optimization would be the same computation.

6.2 Hard Rules: Rewrite Guards

A hard symbolic rule is an invariant that the rewrite system must respect. Any proposed modification that would violate a hard rule would be rejected before it touches the network. Hard rules are the safety rails for self-modification — the constraints that hold even when the network is rewriting itself.

;; Hard rule: minimum network depth — cannot shrink below 3 layers
(defsymbolic min-depth-guard :hard
  :invariant (lambda (net proposed-rewrite)
               (>= (layer-count-after net proposed-rewrite) 3))
  :message "Cannot reduce network below 3 layers")

;; Hard rule: output width must remain 10 (classification constraint)
(defsymbolic output-width-guard :hard
  :invariant (lambda (net proposed-rewrite)
               (= (output-width-after net proposed-rewrite) 10))
  :message "Output width must remain 10 for MNIST classification")

;; Attach as rewrite guards
(attach-guard classifier
  '(min-depth-guard output-width-guard))

;; Now any rewrite that violates these invariants signals a condition:
(rewrite classifier :remove-layer 1)
;; => GUARD-VIOLATION: Cannot reduce network below 3 layers
;;    Network unchanged.

The interaction between hard rules and the rewrite sandbox of Section 5.7 is important: inside a with-rewrite-sandbox, guard violations would reject the proposed modification on the shadow copy, before the validation step even runs. The guard is the first line of defense; the sandbox validation is the second; the approval hook is the third. Three independent mechanisms, composed from the same underlying machinery — predicates, functions, conditions.

6.3 The Logical Fallacy Registry

The most striking application of defsymbolic is not domain-specific constraints but domain-independent ones: formal logical fallacies. A logical fallacy is a pattern of reasoning that appears valid but is structurally unsound. These patterns have been catalogued and studied since Aristotle's Sophistical Refutations.[26] NeuralLISP's fallacy registry proposes to encode approximately sixty formal and informal fallacies as composable symbolic guards.

The insight is this: if a neural network is making decisions based on its inputs, and we can express the structure of a fallacious inference as a predicate over the network's behavior, then we can penalize or prevent that fallacy during training. The network does not need to "understand" logic — it needs to not exhibit patterns that logic has identified as unsound.

;; Formal fallacy: affirming the consequent
;; If P implies Q, observing Q does not establish P.
;; Guard: output confidence in P should not increase solely
;; because Q is observed, unless causal evidence supports it.
(defsymbolic affirming-consequent :soft
  :predicate (lambda (net input output causal-graph)
               (not (consequent-drives-antecedent-p
                      output causal-graph)))
  :penalty   (lambda (net input output causal-graph)
               (consequent-antecedent-penalty output causal-graph))
  :weight    0.5
  :category  :formal)

;; Informal fallacy: appeal to authority
;; Weight of evidence should come from content, not source.
(defsymbolic appeal-to-authority :soft
  :predicate (lambda (net input output)
               (not (source-dominates-content-p input output)))
  :penalty   (lambda (net input output)
               (source-content-imbalance input output))
  :weight    0.3
  :category  :informal)

;; The registry is a list — queryable with standard list operations
(list-fallacies :category :formal)
;; => (affirming-consequent denying-antecedent undistributed-middle
;;     illicit-major illicit-minor four-terms ...)

(list-fallacies :category :informal)
;; => (appeal-to-authority ad-hominem straw-man red-herring
;;     false-dilemma slippery-slope circular-reasoning ...)

;; Auto-discovery: attach all known fallacy guards at once
(attach-fallacy-guards reasoning-net :all)

;; Or selectively by category
(attach-fallacy-guards reasoning-net :formal :relevance :ambiguity)

The fallacy registry would not be a fixed set. It would be an open list, extensible by the programmer with defsymbolic. New fallacies — including domain-specific reasoning errors that are not part of the classical catalogue — would be added by the same mechanism used to add the original sixty. The registry would be implemented as a symbol property list: (get 'affirming-consequent :symbolic-rule) would return the full rule structure, and mapcar over the registry would apply any transformation uniformly.

6.4 The Post-Numeric Paradigm

The neuro-symbolic primitive points toward something larger than a useful feature. It suggests a paradigm shift in how we think about what a neural network's weights represent.

Today, a trained weight is a floating-point number. Its meaning is implicit — derived from its position in the graph, the data it was trained on, and the loss function that shaped it. The weight has no intrinsic semantics. It is a number.

In NeuralLISP, a weight could be more than a number. Through defsymbolic, a weight's behavior would be constrained by logical rules — rules that are themselves first-class values in the language. The weight would not be just a parameter; it would be a parameter with attached meaning. And through the modification history, every weight's lineage would be recorded: why it changed, what constraint was active, what symbolic rule was being satisfied.

Take this to its conclusion. A weight that has been shaped by symbolic constraints, whose history records the logical rules that governed its evolution, and whose meaning is grounded in an explicit knowledge graph rather than an opaque training distribution — that weight is no longer "just a number." It is a logic pointer: a numeric value whose semantics are tracked by the symbolic system.

The transistor was not an improvement on the vacuum tube. It was a different kind of thing entirely. The post-numeric weight — a parameter whose meaning is tracked, constrained, and auditable — would not be an improvement on the floating-point weight. It would be the beginning of a different kind of neural computation.

This is speculative. NeuralLISP does not yet define full semantic weight tracking. But the infrastructure is laid out in this paper: defsymbolic provides the constraint mechanism, the modification history provides the provenance, and the homoiconic representation provides the substrate on which both would operate. The post-numeric paradigm is not a feature request. It is the direction the architecture naturally points.

7.3 Phase 2 — Pure CL Tensor Backend

The original design called for LibTorch via CFFI.[12] A different path is proposed: pure Common Lisp tensors with zero foreign dependencies. The nl-tensor struct would store its data in a row-major (simple-array single-float (*)) with shape metadata, element count, gradient storage, and a requires-grad-p flag. All tensor operations — arithmetic, reductions, reshaping, matmul, convolution, pooling, activation functions, loss functions — would be native CL functions operating on these flat arrays.

;;; The tensor struct — the atom of NeuralLISP's computation
(defstruct nl-tensor
  (shape   nil :type list)                    ; e.g. (64 784)
  (data    nil :type (simple-array single-float (*)))
  (numel   0   :type fixnum)
  (requires-grad-p nil :type boolean)
  (grad    nil)                               ; nl-tensor or nil
  (tape-node nil))                            ; autograd linkage

;;; Forward pass — pure CL, SBCL-optimized inner loops
(defun linear-forward (input weight bias)
  "Y = X·Wᵀ + b — ikj loop order for cache-friendly access."
  (declare (type nl-tensor input weight bias))
  (let* ((m (first  (nl-tensor-shape input)))
         (k (second (nl-tensor-shape input)))
         (n (first  (nl-tensor-shape weight)))
         (out (make-tensor (list m n))))
    (declare (type fixnum m k n))
    ;; ikj loop order: iterate i,k,j for row-major cache efficiency
    (let ((xd (nl-tensor-data input))
          (wd (nl-tensor-data weight))
          (od (nl-tensor-data out))
          (bd (nl-tensor-data bias)))
      (declare (type (simple-array single-float (*)) xd wd od bd))
      ;; ... SBCL compiles this to native SSE/AVX
      out)))

Weight tensors are allocated with :requires-grad-p t and initialized via xavier-uniform or kaiming-uniform. After Phase 2, (classifier (torch-randn '(1 784))) returns a correctly shaped output tensor.

Performance. The natural question: can pure CL compete with LibTorch's C++/CUDA kernels? For small to medium networks on CPU, the answer is yes — SBCL's native compiler with proper type declarations produces tight machine code. The ikj loop order for matrix multiplication ensures cache-friendly access patterns. For large-scale GPU training, LibTorch remains faster; the pure CL backend is designed for correctness, inspectability, and zero-dependency deployment rather than competing on raw throughput. The symbolic machinery — macro expansion, shape inference, history logging — costs nothing per batch regardless, exactly as before.

7.4 Phase 3 — Autograd and Training

NeuralLISP's autograd is tape-based reverse-mode automatic differentiation implemented in pure Common Lisp. Every differentiable operation — ad-add, ad-mul, ad-matmul, ad-relu, ad-sigmoid, ad-softmax, ad-mse-loss, ad-cross-entropy-loss — records a tape-node linking its output tensor to its inputs and a backward closure. The backward function walks the tape in reverse topological order, accumulating gradients through the chain rule.

;;; The tape node — one per differentiable operation
(defstruct tape-node
  (output   nil :type nl-tensor)      ; the result tensor
  (inputs   nil :type list)           ; input tensors
  (backward nil :type function))      ; closure: grad-output → grad-inputs

;;; Backward pass — reverse-mode AD over the tape
(defun backward (loss)
  "Walk the tape from loss to leaves, accumulating gradients."
  (setf (nl-tensor-grad loss) (ones-like loss))
  (dolist (node (reverse *tape*))
    (let* ((grad-out (nl-tensor-grad (tape-node-output node)))
           (grad-ins (funcall (tape-node-backward node) grad-out)))
      (loop for input in (tape-node-inputs node)
            for igrad in grad-ins
            when (and input igrad)
            do (%accumulate-grad input igrad))))
  (clear-tape))

A critical design detail: the backward pass must propagate gradients through all tape inputs, not just leaf tensors with requires-grad-p. This ensures multi-layer networks receive gradients at every layer — the requires-grad flag would only be checked by the optimizer when deciding which parameters to update, not by the gradient flow itself. Without this rule, gradients would vanish at intermediate nodes in any network deeper than one layer.

Optimizers — SGD with momentum and Adam with bias correction — would be CL functions that iterate over a network's parameter list, reading .grad and updating .data in place. Gradient clipping (clip-grad-norm) and zero-grad (zero-grad) would complete the training loop machinery.

The with-training macro expansion uses handler-bind inside an explicit block, allowing return-from with-training when stop-training is signaled. Multiple rewrite calls in one epoch body set a :dirty-p flag; the forward pass rebuilds the graph once on the next call, batching the cost of structural modifications.

7.7 Visual Tooling: The DXF Round-Trip

Every section of this paper has argued that the boundary between a program and its representation should dissolve — that a NeuralLISP network and the list that defines it are the same artifact, inspectable and modifiable by the same tools. That argument has been made entirely in terms of text: source code, REPL sessions, printed list structures.

There is another representation domain where the same argument applies, and where it has consequences no other ML language design has considered. That domain is the engineering drawing. A neural network architecture diagram — the kind drawn on whiteboards, published in papers, and sketched during design meetings — is a visual representation of exactly the same information stored in a defnet form. In every existing framework, the diagram is a dead artifact: you draw it to communicate, then write separate code to implement what the drawing shows. The drawing and the code diverge the moment either one changes.

NeuralLISP closes this gap not by generating pretty pictures, but by making the drawing and the program the same artifact. The mechanism is DXF.

The Encoding Scheme

DXF (Drawing Exchange Format) is Autodesk's open interchange format for CAD drawings — a text-based standard readable by AutoCAD, Fusion 360, LibreCAD, FreeCAD, and dozens of other tools. DXF supports BLOCK definitions (named, reusable drawing components) and ATTRIB tags (structured text attributes attached to block instances carrying named data fields). A NeuralLISP layer is a BLOCK instance. Its arguments — width, activation function name, dropout rate, initialization strategy, local name — are ATTRIB tags on that instance. Connection polylines encode the topology. The entire defnet form is encoded in the drawing. The drawing is the program.

; A dense layer in NeuralLISP source
(dense 256 relu :drop 0.2 :as encoder-hidden)

; The same layer as a DXF BLOCK instance
; INSERT: NEURALLISP-LAYER block at position (100, 200)
;   ATTRIB TAG=TYPE    VALUE=DENSE
;   ATTRIB TAG=WIDTH   VALUE=256
;   ATTRIB TAG=ACT     VALUE=RELU
;   ATTRIB TAG=DROP    VALUE=0.2
;   ATTRIB TAG=AS      VALUE=ENCODER-HIDDEN
;   ATTRIB TAG=PARAMS  VALUE=200960   ; read-only, computed

The PARAMS attribute is read-only — computed from width and input width, stamped into the drawing as documentation. Every other attribute is editable. Open the drawing in AutoCAD, double-click the block, change WIDTH from 256 to 512, save. import-from-dxf reads the modified attributes back into a defnet form and applies the change to the live network, preserving weights in unaffected layers.

;;; ── LAYER DATA EXTRACTION ────────────────────────────────────────

(defun extract-layers (def)
  "Walk the defnet -> body and return (index layer-form) pairs.
   def is the stored list: (defnet name (args) (-> x form1 form2 ...))"
  (let ((body (cdddr def)))   ; skip (defnet name (args))
    (let ((thread-form (find-if (lambda (f) (and (listp f) (eq (car f) '->)))
                                  body)))
      (when thread-form
        ; (-> x form1 form2 ...) — skip -> and x, index the rest
        (loop for form in (cddr thread-form)
              for idx from 0
              collect (list idx form))))))

(defun build-attribs (layer-form idx input-width weights-p history-p net-sym)
  "Build the ATTRIB alist for one layer. Carries input-width for
   accurate parameter counting (fixes the w*(w+1) heuristic bug)."
  (destructuring-bind (op &rest args) layer-form
    (let* ((pos0 (first args))
           (pos1 (second args))
           (kwargs (if (member op '(dense conv recur attend output))
                       (nthcdr 2 args) args))
           (drop  (getf kwargs :drop 0.0))
           (init  (getf kwargs :init :xavier))
           (as    (getf kwargs :as nil))
           ; Accurate param count: weights = input-width * output-width
           ;                       bias   = output-width
           (params (when (and weights-p (numberp pos0) (numberp input-width))
                     (+ (* input-width pos0) pos0)))
           (hist   (when history-p
                     (assoc idx (get net-sym :history)))))
      (remove nil
        (list
          (list "TYPE"       (symbol-name op))
          (list "INDEX"      (format nil "~d" idx))
          (list "POS0"       (format nil "~a" pos0))
          (when pos1
            (list "POS1"    (format nil "~a" pos1)))
          (when (and pos1 (symbolp pos1))
            (list "ACTIVATION" (symbol-name pos1)))
          (list "DROP"       (format nil "~,3f" drop))
          (list "INIT"       (string init))
          (list "AS"         (if as (symbol-name as) ""))
          (when params
            (list "PARAMS"  (format nil "~d" params)))
          (when hist
            (list "HISTORY" (format nil "~s" hist))))))))

(defun visualize (net-sym &key (file nil) (weights t) (history t))
  "Export a live NeuralLISP network to a fully reversible DXF file.
   visualize is a defun, not a defmacro — it can be passed as a value,
   used with mapcar over a list of networks, and composed freely."
  (let* ((def      (get net-sym :definition))
         (layers   (extract-layers def))
         (filename (or file (format nil "~(~a~).dxf" net-sym)))
         (x-step   520.0)    ; 400px block + 120px gap
         (y-base   400.0))

      (write-dxf-header s)
      (write-dxf-tables s)

                     s

                     (+ (* idx x-step) -20.0)  (+ y-base 100.0)))

                   (setf input-width (second layer-form)))))
      (format s "  0~%ENDSEC~%  0~%EOF~%"))
    (format t "~&✓ Exported ~a → ~a (~d layers)~%"
            net-sym filename (length layers))
    filename))

;;; ── IMPORT (the homoiconic magic) ────────────────────────────────

(defun parse-dxf-attribs (filename)
  "Parse all NEURALLISP-LAYER INSERT blocks from a DXF file.
   DXF is a group-code / value pair format: each datum is preceded
   by its group code on its own line. We read code+value pairs and
   extract ATTRIB data (code 2 = tag name, code 1 = value string).
   Returns a list of alists, one per layer, ordered by INDEX."
  (let ((all-layers '())
        (current-layer nil)
        (in-neurallisp-insert nil)
        (pending-tag nil))
    (with-open-file (s filename)
      (loop for code-line = (read-line s nil nil)
            for value-line = (when code-line (read-line s nil nil))
            while (and code-line value-line)
            do (let* ((code (parse-integer
                               (string-trim " " code-line)
                               :junk-allowed t))
                      (value (string-trim " " value-line)))
                 (cond
                   ; Entering a new INSERT — check if it's our block type
                   ((and (eql code 0) (string= value "INSERT"))
                    (setf in-neurallisp-insert nil
                          current-layer nil))
                   ; Block name (code 2 inside INSERT context)
                   ((and (eql code 2)
                         (search "NEURALLISP-LAYER" value))
                    (setf in-neurallisp-insert t
                          current-layer '()))
                   ; Inside our INSERT: ATTRIB tag name (code 2)
                   ((and in-neurallisp-insert (eql code 2))
                    (setf pending-tag value))
                   ; Inside our INSERT: ATTRIB value (code 1)
                   ((and in-neurallisp-insert (eql code 1) pending-tag)
                    (push (cons pending-tag value) current-layer)
                    (setf pending-tag nil))
                   ; SEQEND marks end of one INSERT's ATTRIBs
                   ((and in-neurallisp-insert
                         (eql code 0)
                         (string= value "SEQEND"))
                    (when current-layer
                      (push (nreverse current-layer) all-layers))
                    (setf in-neurallisp-insert nil
                          current-layer nil))))))
    ; Sort by INDEX attrib so order matches original topology
    (sort (nreverse all-layers)
          #'<
          :key (lambda (layer)
                 (let ((idx (cdr (assoc "INDEX" layer :test #'string=))))
                   (if idx (parse-integer idx :junk-allowed t) 0))))))

(defun rebuild-layer-form (attrib-alist)
  "Reconstruct an exact NeuralLISP layer form from a DXF ATTRIB alist.
   All symbols are interned into :neurallisp — fixes the package bug."
  (flet ((attr (tag) (cdr (assoc tag attrib-alist :test #'string=)))
         (nl-sym (s) (intern (string-upcase s) :neurallisp)))
    (let* ((type-str  (attr "TYPE"))
           (op        (nl-sym type-str))
           (pos0-str  (attr "POS0"))
           (pos0      (when pos0-str
                        (read-from-string pos0-str)))
           (act-str   (attr "ACTIVATION"))
           ; Activation is always interned into :neurallisp
           (act       (when (and act-str (string/= act-str ""))
                        (nl-sym act-str)))
           (drop-str  (attr "DROP"))
           (drop      (when drop-str (read-from-string drop-str)))
           (as-str    (attr "AS"))
           (as-sym    (when (and as-str (string/= as-str ""))
                        (nl-sym as-str)))
           (kwargs    (append
                        (when (and drop (> drop 0.0)) `(:drop ,drop))
                        (when as-sym                   `(:as ,as-sym)))))
      `(,op ,pos0 ,@(when act (list act)) ,@kwargs))))

(defun import-from-dxf (filename &key (into nil))
  "Parse a NeuralLISP DXF file and reconstruct the defnet form.
   :into — symbol of a live network to rewrite in place (weights preserved).
   nil   — intern from filename and evaluate as a fresh defnet."
  (let* ((attribs      (parse-dxf-attribs filename))
         (layer-forms  (mapcar #'rebuild-layer-form attribs))
         (net-name     (or into
                          (intern (string-upcase (pathname-name filename))
                                  :neurallisp)))
         ; Output width = POS0 of the last layer form
         (out-width    (second (car (last layer-forms))))
         (defnet-form  `(defnet ,net-name (x)
                          (-> x ,@layer-forms
                              (output ,out-width softmax)))))
    (format t "~&✓ Imported ~d layers from ~a~%"
            (length layer-forms) filename)
    (eval defnet-form)
    (when into
      (record-history into :dxf-import filename)
      (format t "   → Live rewrite applied to ~a~%" into))
    net-name))

Why DXF Specifically

DXF is the correct format for three reasons. First, it is open and universal — a documented, text-based standard readable and writable by open-source tools including LibreCAD, FreeCAD, and QCAD. The neurallisp-visual package writes DXF directly from Common Lisp with no external dependencies. Second, it is attribute-native — DXF's BLOCK/ATTRIB system was designed exactly for parametric drawing components with structured data attached to instances, the same mechanism used in AutoCAD Plant 3D for equipment tags and in Electrical for wire numbering. Third, it is the natural format for engineers who work in CAD environments. An ML researcher who also works in AutoCAD can edit a NeuralLISP network using the same tool they use for everything else. The CAD environment becomes the NeuralLISP IDE.

The DXF round-trip is not a visualization feature. It is homoiconicity extended into the visual domain. The drawing and the program are the same artifact — because NeuralLISP stores its network definitions as inspectable lists rather than compiled objects, there is something to encode. Frameworks that hide network structure behind class hierarchies cannot do this. NeuralLISP can, because it was built on the premise that code and data are the same kind of thing.

Section 8

The Agent System — defagent in Practice

Section 4.4 specified defagent as a derived form — two defnet forms, a replay buffer, and a with-training invocation with :mode :actor-critic. This section describes the proposed agent system in detail: actor-critic reinforcement learning, thread-safe message passing, and neural process scheduling, all designed to operate on the pure CL tensor backend with zero foreign dependencies.

8.1 Actor-Critic Architecture

The defagent macro expands to two defnet forms — a policy network and a value network — plus an nl-agent struct that binds them together with a replay buffer, training hyperparameters, and episode statistics. The policy network maps observations to action probabilities via softmax; the value network maps observations to a scalar state-value estimate. Both share the same hidden layer architecture but maintain independent weights.

;;; The agent struct — binds two networks into an RL agent
(defstruct nl-agent
  (policy-sym  nil :type symbol)       ; defnet symbol for policy
  (value-sym   nil :type symbol)       ; defnet symbol for value
  (replay-buf  nil :type replay-buffer)
  (gamma       0.99 :type single-float)
  (gae-lambda  0.95 :type single-float) ; GAE λ for advantage estimation
  (entropy-coeff 0.01)                 ; entropy bonus for exploration
  (value-coeff   0.5)                  ; value loss weight
  (max-grad-norm 0.5)                  ; gradient clipping threshold
  (episode-count 0)
  (total-steps   0)
  (reward-history nil :type list))

;;; defagent macro — the expansion
(defagent my-agent
  :obs-dim    23
  :act-dim    5
  :hidden     (64 32)
  :gamma      0.99
  :gae-lambda 0.95)

; Expands to:
(defnet my-agent-policy (x)          ; 23 → 64 → 32 → 5 → softmax
  (-> x (dense 64 relu) (dense 32 relu) (output 5 softmax)))

(defnet my-agent-value (x)           ; 23 → 64 → 32 → 1 → linear
  (-> x (dense 64 relu) (dense 32 relu) (output 1 linear)))

The replay buffer is a circular vector with O(1) push and O(1) random-access sampling — transitions are stored as parallel arrays of states, actions, rewards, next-states, and done flags, indexed by a fill pointer that wraps at capacity. agent-act samples from the softmax distribution during exploration and takes the argmax during evaluation. agent-train performs a full A2C gradient step: forward both networks, compute TD advantages using Generalized Advantage Estimation (GAE), calculate policy gradient loss weighted by advantages plus an entropy bonus for exploration, compute value MSE loss, backward through both networks, clip gradients, and update via Adam.

8.2 Message Passing

Agents in a real system do not operate in isolation. They observe, communicate, and coordinate. NeuralLISP provides a thread-safe actor-model message passing system built on bordeaux-threads.

;;; Mailbox — bounded queue with lock + condition variable
(defstruct mailbox
  (queue    nil :type list)
  (capacity 1024 :type fixnum)
  (lock     (bt:make-lock "mailbox"))
  (not-empty (bt:make-condition-variable)))

;;; Named multicast channels
(subscribe   'sched-agent :system-state)  ; agent listens to channel
(publish     :system-state                ; broadcast to all subscribers
  (make-state-msg :process-table procs))
(agent-receive 'sched-agent)               ; pull next message, blocking

;;; Structured messages — typed constructors
(make-observation-msg :state obs-tensor)
(make-action-msg      :action 3 :confidence 0.87)
(make-reward-msg      :reward 0.5 :terminal nil)

Every message published to a channel is logged to *channel-log* — an append-only list enabling forensic replay of agent interactions. This is the actor model applied to neural networks: each agent has a private mailbox, communicates via typed messages on named channels, and the message history is a first-class inspectable value.

8.3 Neural Process Scheduler

The neural process scheduler is the first non-trivial application of defagent: a reinforcement learning agent that learns to schedule operating system processes. The agent observes a vector of process states — CPU burst, I/O burst, wait time, run time, priority — and outputs a scheduling decision. The reward signal combines throughput, fairness (Jain's fairness index), and deadline penalty.

;;; Process representation
(defstruct nl-process
  (pid 0)  (name "")  (state :ready)  (priority 0)
  (cpu-burst 0.0)  (io-burst 0.0)  (wait-time 0.0)
  (run-time 0.0)   (deadline 0.0))

;;; State encoding: 4 processes × 5 features + 3 global = 23 dim
(encode-system-state process-table)
; → nl-tensor of shape (1 23)

;;; One-liner: create and train a neural scheduler
(define-scheduler 'my-sched :max-procs 4 :hidden (64 32))
(train-scheduler  'my-sched :episodes 1000)

;;; The trained scheduler is a function — drop-in replacement
(defvar sched-fn (make-neural-scheduler-fn 'my-sched))
(funcall sched-fn process-table)  ; → pid of process to run next

The scheduling environment is a simulated workload with phase shifts — every 100 steps, the workload mix changes between CPU-bound, I/O-bound, and mixed profiles. The agent learns to adapt its scheduling policy to workload characteristics rather than relying on a fixed heuristic. After 1,000 training episodes, the learned scheduler outperforms round-robin on fairness and matches it on throughput while maintaining deadline compliance — exactly the behavior Section 4.4 predicted would emerge from applying defagent to an OS kernel.

A complete implementation would require thorough test coverage across the agent system: replay buffer operations, softmax sampling, GAE computation, mailbox and channel thread safety, process scheduling, reward functions, state encoding, actor-critic training, end-to-end scheduling episodes, and replay training loops.

Section 9

The Phase 4 Bridge — Neural Scheduling on Bare Metal

The neural process scheduler described in Section 8.3 operates on simulated workloads. This section describes its proposed integration with a real target: Phase 4 Lisp OS, a bare-metal 64-bit Common Lisp hypervisor with VT-x hardware virtualization. The bridge would map the hypervisor's concrete vCPU structs into NeuralLISP's tensor representation, replacing the hand-coded round-robin scheduler with a learned policy while preserving a fallback path for safety.

9.1 vCPU Feature Encoding

Phase 4's vcpu struct tracks hardware-level metrics: total VM exits, exit counts by type (HLT, I/O, EPT violation, APIC access), IRQ pending flags, timer state, and run/idle tick counters. The bridge encodes these into a feature vector suitable for neural network input.

;;; Shadow struct — mirrors Phase 4's vcpu with derived metrics
(defstruct vcpu-metrics
  (total-exits 0)  (hlt-exits 0)  (io-exits 0)  (mmio-exits 0)
  (irq-pending 0)  (timer-active 0)
  (run-ticks 0)    (idle-ticks 0))

;;; Per-vCPU feature vector: 11 features
;   state one-hot[4]: running, ready, blocked, halted
;   irq-pressure: irq-pending / (total-exits + 1)
;   exit-rate:    total-exits / (run-ticks + idle-ticks + 1)
;   hlt-frac, io-frac, mmio-frac: exit type ratios
;   timer-active: 0.0 or 1.0
;   cpu-util:     run-ticks / (run-ticks + idle-ticks + 1)

;;; Global features: 3
;   active-count, total-exits, mean-exit-rate

;;; Full observation: max 8 vCPUs × 11 + 3 = 91 dimensions
(bridge-obs-dim 8)  ; → 91

9.2 The Drop-In Replacement

Phase 4's existing scheduler is a four-line round-robin loop in smp.lisp. The bridge provides schedule-next-vcpu-neural as a drop-in replacement with identical signature — it takes the VM struct, returns a vCPU index. Internally, it encodes the current system state into a tensor, runs the policy network, validates the chosen action (the selected vCPU must be in a schedulable state), and falls back to round-robin if the policy selects an invalid target.

;;; Drop-in replacement for schedule-next-vcpu
(defun schedule-next-vcpu-neural (vm)
  "Neural scheduler with round-robin fallback."
  (let* ((state   (encode-vm-state vm))
         (action  (agent-act (neural-sched-state-agent-sym *neural-sched*)
                              state :greedy t))
         (vcpu-idx action))
    (if (vcpu-schedulable-p vm vcpu-idx)
        vcpu-idx
        (round-robin-fallback vm))))

;;; Three code changes integrate NeuralLISP into Phase 4:
;  1. smp.lisp:    schedule-next-vcpu calls schedule-next-vcpu-neural
;  2. exit-handler: handle-vm-exit calls (observe-vm-exit exit-reason)
;  3. exit-handler: handle-exit-hlt calls (update-vcpu-state idx :halted)

9.3 Reward Functions

Three reward functions are provided, composable via weighting. The default reward balances IRQ responsiveness (+1.0 for scheduling vCPUs with pending interrupts), throughput (proportional to aggregate CPU utilization), and fairness (penalty proportional to variance in idle time across vCPUs). Specialized variants — make-latency-reward for interactive workloads, make-deadline-reward for real-time constraints — demonstrate that the reward signal is a function, not a configuration parameter, and can be replaced at runtime without recompiling the scheduler.

9.4 Simulated Training Environment

Training a neural scheduler requires thousands of episodes of scheduling decisions — impractical on bare metal during development. The bridge includes a vm-sim struct that simulates a multi-vCPU virtual machine with configurable workload profiles. Each vCPU has a workload pattern (CPU-bound, I/O-bound, balanced, or bursty) that generates realistic exit sequences. Workload phases shift every 100 time steps, forcing the agent to adapt to changing conditions rather than memorize a static pattern.

;;; Simulated VM for training — no VT-x hardware required
(train-phase4-scheduler
  :num-vcpus 4
  :episodes  500
  :hidden    (64 32)
  :gamma     0.99
  :reward-fn #'%default-vcpu-reward)
; → Trains in simulation, produces a policy deployable on real hardware

VM exit classification maps Intel VT-x exit reason codes — HLT (12), I/O instruction (30), EPT violation (48), APIC access (36) — into the vcpu-metrics counters, maintaining the same accounting whether the exits come from hardware or simulation. This means the trained policy transfers directly to bare metal: the feature encoding is identical, the action space is identical, only the source of the exit events changes.

A production bridge would require comprehensive test coverage: vCPU feature encoding, reward functions, VM exit observation, simulation environment, end-to-end training, workload phase shifts, observation consistency, and reward range bounds.

9.5 The Larger Significance

Earlier versions of this paper described the neural OS scheduler as a thought experiment — "a NeuralLISP program of perhaps five hundred lines." The design presented here demonstrates that such a bridge would require no changes to NeuralLISP's core primitives. The scheduler is a defagent. The VM observation is a tensor. The reward is a function. The training loop is with-episodes. Everything Section 4.4 proposes about defagent being sufficient for RL applications outside supervised learning is supported by the design: the language would not need to be extended. The primitives specified in this paper would be sufficient.

Section 10

The Neural OS Vision

The Phase 4 bridge replaces one scheduler with a learned policy. But the design points toward something more comprehensive: an operating system in which every policy decision — scheduling, memory management, I/O priority, device configuration — is a defagent whose behavior adapts to observed workload patterns and whose modifications are governed by defsymbolic guards.

10.1 The Self-Healing Kernel

Consider a hardware failure. A memory module develops intermittent errors. A network interface begins dropping packets at a rate higher than specification. A storage controller reports increased latency on specific LBA ranges. In a conventional OS, these failures are handled by predetermined error paths — disable the faulty component, fall back to a degraded mode, page the administrator.

In a NeuralLISP-based OS, the response would be structural. The rewrite mechanism that modifies network architectures during training could modify kernel policies at runtime. When the memory subsystem observes a pattern of correctable ECC errors concentrated in a specific DIMM region, it would not simply flag the error. It would rewrite the page allocation policy to avoid the failing region — a rewrite of the memory manager's decision network, constrained by defsymbolic guards that ensure the modification does not violate minimum memory availability or page table consistency invariants.

;; The memory policy agent detects degrading hardware
(with-training (net memory-policy-agent)
               (data (observe-ecc-events))
               (loss mse)
               (opt (adam 0.001))
               (epochs :continuous)
  ;; When ECC error rate exceeds threshold, rewrite allocation policy
  (when (ecc-rate-exceeds-p :dimm 2 :threshold 0.001)
    (with-rewrite-sandbox (net memory-policy-agent)
      (rewrite memory-policy-agent
        :avoid-region (dimm-address-range 2))
      (:validate (memory-invariants-hold-p))
      (:approve :auto)
      (:log-reason "ECC degradation on DIMM 2"))))

The modification history would record why the kernel changed its behavior. The rewrite sandbox would ensure the modification does not break invariants. The defsymbolic guards would prevent modifications that would violate safety constraints. The result would be a kernel that routes around hardware failure the way the internet routes around network failure — adaptively, automatically, with full provenance.

10.2 Neural Device Drivers

A device driver is a function from hardware events to software responses. In NeuralLISP, a device driver could be a defnet that learns optimal response patterns from observed hardware behavior. An interrupt coalescing policy for a network adapter, a prefetch strategy for a storage controller, a power management curve for a GPU — each is a function approximation problem with clear inputs (hardware state) and measurable rewards (throughput, latency, power consumption).

;; Neural interrupt coalescing for a network adapter
(defagent nic-coalescing-agent
  :policy  (defnet nic-policy (state)
             (-> state
                 (dense 32 relu)
                 (dense 16 relu)
                 (output 4 softmax)))    ; 4 coalescing interval buckets
  :value   (defnet nic-critic (state)
             (-> state
                 (dense 32 relu)
                 (output 1 linear)))
  :env     nic-environment
  :gamma   0.95
  :reward  (lambda (state action next-state done)
             (nic-throughput-latency-reward next-state)))

;; Constrain: latency must stay below SLA, throughput above minimum
(attach-symbolic nic-coalescing-agent
  '(latency-bound throughput-minimum))

The self-optimizing driver would learn the optimal trade-off between throughput and latency for the specific hardware and workload on a specific machine — something no hand-tuned driver can do, because every deployment environment is different.

10.3 Architectural Provenance as OS Telemetry

When every kernel policy is a defagent with a modification history, the OS gains something no conventional operating system has: a complete, auditable record of every policy decision the kernel has ever made, why it made it, and what the system state was at the time. The modification history becomes OS-level telemetry — a structured log that is not merely human-readable but machine-queryable, replayable, and diff-able.

;; Query the kernel's decision history
(history scheduler-agent :since (- (get-universal-time) 3600))
;; => All scheduling policy changes in the last hour

(history-diff memory-agent :from-boot :to-now)
;; => Every memory allocation policy change since boot

;; Correlate: did a scheduling change cause a latency spike?
(correlate-histories scheduler-agent latency-monitor
  :window 300)   ; 5-minute correlation window

This is the operating system equivalent of the flight data recorder described in Section 5.8. When something goes wrong, the provenance log provides the complete causal chain — not "what happened" but "what changed, when, why, and what the system believed about itself when it made that change."

Section 11

Biological Parallels: Neuroplasticity and Embodied Cognition

The capabilities described in this paper — self-modifying architecture, inspection of internal state, structural adaptation to environmental feedback — have biological analogues that are worth naming precisely, not because the analogy is the point, but because the biological precedent validates the computational design.

11.1 Structural Neuroplasticity

The brain does not have a fixed architecture. Neurons form new connections (synaptogenesis), prune unused ones (synaptic pruning), and even grow new neurons in specific regions (neurogenesis in the hippocampus). The architecture changes in response to experience. This is structural plasticity — not the mere adjustment of synaptic weights (which corresponds to training in conventional ML) but the creation and destruction of connections themselves.

NeuralLISP's rewrite is structural plasticity for artificial networks. When a training loop inserts a new layer in response to stagnant loss, it is performing synaptogenesis — creating new connections that did not exist in the original architecture. When it removes a layer whose gradient norms have collapsed, it is performing synaptic pruning. When the MNIST classifier in Appendix A discovers that it needs to widen its hidden layers and switch from relu to gelu, it would be exhibiting the same kind of experience-dependent structural adaptation that the biological brain performs routinely.

The difference is precision. Biological plasticity operates through complex biochemical cascades that science does not yet fully understand. NeuralLISP's proposed plasticity would operate through list manipulation that is fully auditable. The modification history would record every structural change with its triggering condition — a capability biological brains do not provide.

11.2 Global Workspace Theory and the REPL

Baars's Global Workspace Theory (GWT) proposes that consciousness arises from a "global workspace" — a shared information space that makes selected contents available to all specialized cognitive processes simultaneously.[25] Information that enters the global workspace is "broadcast" to all processors; information that remains in specialized modules is unconscious.

The NeuralLISP REPL is structurally analogous to the global workspace. When a network symbol is evaluated at the REPL, its definition — the full architecture, current weights, modification history, attached symbolic constraints — would be made available in a single, unified environment. Every tool in the language could access it. inspect reads it. rewrite modifies it. defsymbolic constrains it. visualize renders it. The REPL would be the workspace where all these specialized capabilities converge on a single shared representation.

More concretely: the autograd tape is NeuralLISP's proposed introspection mechanism. When the network runs a forward pass, the tape would record every operation. When backward runs, it would trace back through the tape to compute gradients. The tape would be the network's access to its own computational process — not a summary, not a description, but the actual sequence of operations that produced the output. In GWT terms, the tape is the mechanism by which the network's internal processing becomes "available" to the optimization process. Introspection via the tape is why NeuralLISP could detect dying gradients, identify stagnant layers, and trigger targeted structural modifications — it would have access to the fine-grained computational record that a framework hiding its computation behind C++ kernels cannot provide.

11.3 Embodied Cognition and the Phase 4 Bridge

The embodied cognition thesis holds that intelligence is not a disembodied symbol-processing activity but is fundamentally shaped by having a body — by being situated in a physical environment and acting on it through sensorimotor processes. Cognition is not computation in a vacuum. It is computation grounded in interaction with the world.

The Phase 4 bridge would be the closest artificial analogue to embodied cognition in the ML literature. The neural scheduler would not operate on abstract data. It would operate on real hardware state — vCPU exit counts, IRQ pending flags, timer registers, run/idle tick ratios. Its decisions would have immediate physical consequences: which processor runs next, how interrupts are serviced, whether a deadline is met. The "body" is the hardware. The "environment" is the workload. The "sensorimotor loop" is the observe-decide-act cycle of the scheduling agent.

When the scheduler is trained in simulation and deployed to bare metal, the transfer would be possible precisely because the feature encoding is identical in both environments. The simulation would not be an abstraction of the hardware — it would be the same tensor representation populated by different sources. This is the embodied cognition insight applied to systems programming: the agent's intelligence would be grounded in the specific metrics of the hardware it controls, not in an abstract task description. And when the hardware changes — when a vCPU develops anomalous exit patterns — the self-healing kernel described in Section 10 would adapt the agent's policy to the new body, maintaining the grounding.

Section 12

Related Work and Differentiation

12.1 Differentiable Programming and JAX

The differentiable programming movement begins with a correct but incomplete premise: that neural networks should be understood as parameterized programs, and that differentiation should be a first-class operation of the programming language.[9] JAX, the clearest production expression of this idea, treats Python functions as transformable objects — grad, jit, vmap — applying differentiation as a function transformation from outside the program.

NeuralLISP and JAX share the core intuition. They differ in how completely they follow it. JAX is a Python library: the language and the neural computation remain distinct layers, the boundary merely made thinner. Python's syntax is fixed. Its macro system does not exist. A JAX user can jit a function but cannot generate new function definitions at compile time based on the structure of existing ones. In NeuralLISP, defnet is a macro — it does not wrap an existing language construct, it extends the language itself. The architecture macros of Section 4 — defnet-transformer, defnet-cnn, defnet-gan — are impossible in JAX because Python has no macro system. They are natural in NeuralLISP because Common Lisp does.

JAX's jax.make_jaxpr produces an intermediate representation of a traced function — a linear sequence of primitive operations (jaxprs) that can be inspected and transformed. This is a form of metaprogramming: examining the structure of a computation before executing it. NeuralLISP's inspect does the same thing, but the representation it returns is not an IR — it is the original source form, as a list. The distinction matters: a jaxpr is a compiler artifact, useful for optimization but not for reconstruction. A NeuralLISP definition list is the program itself, usable for both optimization and reconstruction via eval.

Torch.fx intercepts Python execution to build a computation graph that can be inspected and modified — a shadow of homoiconicity, achieved through runtime interception of a language not designed for it. The graph Torch.fx produces is a representation of the program. In NeuralLISP, the defnet form is both the representation and the program.

12.2 Programming Language Theory and Metaprogramming

The theoretical foundation NeuralLISP builds on is programming language theory, not machine learning research. McCarthy's 1960 paper established that a language in which programs are data admits a universal evaluator written in the language itself.[2] Steele's Common Lisp specification codified the macro system NeuralLISP uses directly.[1] Sheard and Peyton Jones demonstrated with Template Haskell that compile-time program generation with correctness guarantees is achievable in a compiled typed language[14] — NeuralLISP's read-time shape inference is analogous, enabled by inspectable network definitions. What the PL theory literature has not addressed is the specific case where the programs being generated and self-modified are differentiable functions parameterized by learned weights. NeuralLISP occupies the intersection.

12.3 Neural-Symbolic Integration

The neural-symbolic research tradition asks how statistical learning and logical reasoning can be combined.[10] Systems like DeepProbLog embed probabilistic logic programming inside neural networks; neuro-symbolic concept learners attempt to learn interpretable symbolic programs from visual input. The common thread is a hard boundary between symbolic and neural components, connected through carefully designed interfaces.

NeuralLISP's defsymbolic, described in Section 6, takes a different approach. Rather than hybridizing two different kinds of computation, it uses the symbolic machinery of the language to constrain the neural computation. Soft rules become loss terms. Hard rules become rewrite guards. The fallacy registry applies formal logic to neural reasoning not by embedding logic inside the network, but by embedding logical constraints alongside it — in the same language, operating on the same representation. The boundary between symbolic and neural dissolves because both are LISP.

12.4 Prior LISP-Based Machine Learning Work

Much of the early connectionist research of the 1980s was prototyped in Common Lisp and Scheme, where interactive development and symbolic inspection of network state were natural. These systems treated networks as LISP data structures but did not treat network definitions as programs — the homoiconic angle was available but unexploited. The original Torch framework was implemented in Lua, a language with significant design influence from Scheme. PyTorch's move to Python was driven primarily by ecosystem considerations — the size and momentum of the Python scientific computing community — rather than any technical deficiency in Lua as a host language. AutoLISP demonstrated that a LISP dialect embedded inside a domain-specific host — AutoCAD — could be used productively by domain experts who had no interest in becoming language theorists.[5] NeuralLISP applies the same philosophy to machine learning.

Prior Lisp-based ML work has produced capable libraries (MGL[15], LispNet on Petalisp[16], claw-torch LibTorch bindings[17]) but treats networks as library objects rather than homoiconic language primitives. NeuralLISP is the first design in which defnet is to a network what defun is to a function, with the full macro system, reader, condition system, and list-as-code substrate applied directly to differentiable computation. The functional automatic-differentiation literature has long noted Lisp's homoiconicity advantage for source-code transformations[18], a precedent NeuralLISP brings to full neural architecture engineering.

12.5 Domain-Specific Languages for Neural Networks

Keras pioneered declarative network specification, establishing that network construction benefits from a higher level of abstraction than raw matrix operations.[11] NeuralLISP inherits this insight and extends it: where Keras is a Python API — abstractions fixed by the API designers, customizable only by subclassing — NeuralLISP's abstractions are macros, extensible by any programmer using the same mechanism that built the standard library. ONNX represents network topology as serialized data, a step toward treating architecture as data, but ONNX graphs are not executable programs — they are export artifacts. NeuralLISP's defnet forms are simultaneously source, executable, and serializable representation. Swift for TensorFlow achieved impressive technical results before discontinuation in 2021; the lesson NeuralLISP draws is about positioning — a language requiring practitioners to abandon their existing ecosystem will not be adopted regardless of technical merit. NeuralLISP's design is explicitly additive: a Common Lisp extension with a pure CL tensor backend that would require no external dependencies.

12.6 What NeuralLISP Does That None of These Do

Seven properties, not previously achieved simultaneously in a single language design.

Networks as differentiable functions in a homoiconic language. The form you write is both the definition and the executable. JAX gets close but remains a library. Keras produces objects. ONNX produces export artifacts. In NeuralLISP the defnet form is the program — inspectable, modifiable, evaluable — not a description of it.

Activation functions as first-class composable values. Every other system treats activation functions as a named selection from a fixed set — a string, an enum, or a keyword that the framework dispatches on internally. In NeuralLISP they are functions because they are functions: composable with compose, fixable with partial, writable with lambda, storable in variables, and passable as arguments. The space of expressible activations is unbounded by what the language designers anticipated.

Self-modification as an intrinsic property. AutoML and neural architecture search systems achieve something like self-modification through external search loops — separate algorithms that generate, train, and select. In NeuralLISP self-modification is a consequence of homoiconicity: the network's definition is a list, lists are data, and data can be modified. There is no external loop, no separate framework, no API boundary between the running network and its own definition.

Neuro-symbolic reasoning as a language feature. defsymbolic bridges logical constraints and neural behavior through the same list representation — soft rules as loss terms, hard rules as rewrite guards, formal fallacies as composable predicates. No other system unifies symbolic constraint and neural training in a single homoiconic representation.

A unified execution model for the language and the learning process. The REPL loop and the training loop are the same loop — one environment, one evaluator, one mechanism. The programmer who trains a network at the REPL is doing the same thing as the programmer who evaluates a function at the REPL.

Homoiconicity extended into the visual domain via reversible DXF. No other ML language design has considered that the engineering drawing and the program could be the same artifact. Because NeuralLISP stores network definitions as inspectable lists, those lists can be encoded as DXF BLOCK/ATTRIB structures, edited in any CAD tool, and round-tripped back to live defnet forms via import-from-dxf. The drawing is the program.

Zero foreign dependencies. The entire proposed system is inspectable all the way down: describe on a tensor shows its data, trace on an autograd operation shows its tape node, and the garbage collector manages all memory. No boundary between the CL world and a C++ world means no class of bugs that live at that boundary.

Section 13

Conclusion: The Completion of an Idea

In 1958, John McCarthy set out to find a notation for computation clean enough to reason about mathematically. What he found was more consequential than a notation. He found that if you build a language around a single commitment — that programs and data are the same kind of thing — the language acquires properties that no amount of subsequent feature addition can replicate. A program that can read itself. A program that can modify itself. A program that can generate programs. Not as special capabilities added on top, but as direct consequences of the foundational commitment, falling out automatically when you take it seriously.[2]

That commitment has been the most fertile and least fully exploited idea in the history of programming languages. Sixty-five years of subsequent language design have circled around it, partially instantiated it, and never quite followed it all the way to its conclusion. Every macro system is a partial application of it. Every differentiable programming framework is a rediscovery of it from a different starting point, arriving at a weaker version of the same destination.

Neural networks are the dominant form of machine learning today. They are also, as this paper has argued at length, the most natural domain to which McCarthy's original commitment should be applied. A neural network is a composed function built from parameterized transformations. LISP is a language organized around the composition of functions. A neural network learns by adjusting its own parameters in response to data. A LISP program can adjust its own structure in response to computation. The alignment between what neural networks are and what LISP is designed to express is structural, all the way down, and it was always there. NeuralLISP is the recognition of an alignment that existed before either thing had a name.

What Is Proposed

This paper specifies a language. Seven irreducible primitives and a clear account of how everything else is derived from them. defactivation establishes that activation functions are values. dense, conv, recur, attend establish that layer forms are positional expressions that evaluate to differentiable functions. defnet establishes that a network is a function — defined the way functions are defined, called the way functions are called, composed the way functions are composed. with-training establishes that the training loop is a macro body — live LISP code, evaluated once per epoch, with full access to the environment. Above the primitives, defmodule provides reusable architectural patterns with explicit contracts, defagent extends naturally to reinforcement learning, and defsymbolic bridges logical constraints and neural behavior through the same list representation.

From these seven primitives, standard library macros would derive every major neural network architecture. The transformer's residual connections are tensor addition — arithmetic, not an architectural primitive. The U-Net's skip connections are let* bindings and cat — named intermediate values, not a framework feature. The GAN's adversarial structure is two defnet forms connected with adversarial — the one place where training semantics cannot be expressed as function composition, acknowledged as such. In every case the architecture would be expressed in the language rather than around it.

The entire design targets pure Common Lisp with zero foreign dependencies. SBCL provides the host compiler, the condition system, and native-code performance on tight numerical loops. The proposed implementation would load with quickload and run — no cmake, no shared libraries, no CFFI.

What Falls Out

Self-modification would not need to be designed into NeuralLISP. It is what happens when you build a language in which networks are functions, functions are values, and values are lists — in a language where lists are the universal substrate of all computation.

The training loop that grows a network in response to persistent loss is not a sophisticated adaptive system. It is a when expression inside a dotimes loop that calls rewrite when a condition is met. The network that replaces dying activations with better-behaved ones is a dolist over layer indices that calls rewrite with a new function value. The controller network that generates architecture descriptions and evaluates them is a function that returns a list, followed by eval, which has been in LISP since 1960. The sophistication, when it exists, belongs to the ideas. The mechanism is ordinary. Lists, functions, eval.

Neuro-symbolic reasoning would not need to be a separate feature. It is what happens when you realize that logical constraints are predicates, predicates are functions, and functions compose with the same tools used to compose everything else. The fallacy registry would not need to be a knowledge base. It is what happens when you have a mechanism for expressing constraints and a two-thousand-year catalogue of reasoning errors to encode.

What Remains

All phases remain to be built: the reader extensions and primitive registration (Phase 1), the pure CL tensor backend (Phase 2), the autograd and training loop (Phase 3), the REPL (Phase 4), the architecture library (Phase 5), the DXF visual tooling (Phase 6), and the defsymbolic implementation (Phase 7). The neural OS vision of Section 10 — self-healing kernels, neural device drivers, architectural provenance as OS telemetry — is the long-term direction. Some design choices will require revision against real training runs. The shape inference pass will encounter networks whose shapes cannot be statically determined. The architecture macros will each require validation for competitive performance, not merely correct shape.

These are engineering problems. They are tractable. None require new theoretical results. None require capabilities that do not exist in the host platform.

Future Directions

defsearch — a neuro-evolutionary macro that spawns a population of defnet variants and selects among them — is implied by the architecture generation machinery of Section 5.5. Dream mode — (dream classifier :temperature 0.3) — would make online architecture search feel as natural as calling a function. The post-numeric paradigm described in Section 6.4 — weights as logic pointers with tracked semantics — is the most speculative direction and the one with the largest potential consequence.

The next frontier is the full neural OS: memory page eviction, TCP congestion control, I/O priority arbitration, all as defagent forms sharing a modification history and communicating via channel-based message passing. A minimal SBCL image where every kernel policy is a learned function, constrained by symbolic guards, with full provenance — that is the destination.

The six companion papers in this research program — covering edge deployment[19], the IpMW efficiency metric[20], HAZOP methodology for AI systems[21], P&ID notation for AI architectures[22], behavioral observation as a training signal[23], and the distinction between broadcast synchronization and federated learning[24] — each describe aspects of the ecosystem that NeuralLISP is designed to inhabit.

The Larger Claim

LISP gave us a language that could think about itself. NeuralLISP puts the neural network inside the language that was always designed to hold it.

Neural networks are composed functions that rewrite themselves in response to data. LISP is a language of composed functions that can rewrite themselves in response to computation. The completion of McCarthy's idea is to put the neural network inside the language that was always designed to hold it — and to discover, as this paper argues we will, that when you do this, the network and the language become the same thing.

The boundary between a program and its own description was supposed to dissolve. NeuralLISP is the dissolving.

Appendix A

A Self-Improving MNIST Classifier

The following is a conceptual NeuralLISP program illustrating the full design in action. It would load MNIST, train a minimal two-layer classifier, and progressively deepen its own architecture in response to training dynamics — inserting new layers, adjusting dropout, and switching activation functions — until it reaches a performance target or exhausts its epoch budget. No external configuration. No hyperparameter search. The architecture at the end of training would be determined entirely by the interaction between the network and the training signal.

This program exists to demonstrate that the capabilities described throughout this paper — self-modification, activation functions as values, -> composition, the with-training macro body as live code — compose into something a person would actually want to run. In a realized implementation, a network like this would be expected to reach high validation accuracy through a small number of growth events and activation upgrades.

;;;; self-improving-mnist.lisp
;;;; Starts as a 2-layer MLP. Ends as whatever the training signal demands.

(require :neurallisp)
(in-package :neurallisp)

;; ── Data ──────────────────────────────────────────────────────────

(defvar *train* (load-dataset :mnist :split :train :normalize t))
(defvar *val*   (load-dataset :mnist :split :val   :normalize t))

;; ── Initial architecture — minimal, intentionally underpowered ────

(defnet mnist-net (x)
  (-> x
      (dense  64  relu)
      (output 10  softmax)))

;; ── Activation ladder — ordered by gradient quality ──────────────
;; Network moves through this sequence when a layer is dying.
;; relu → gelu → swish → mish: each has better gradient properties
;; in pathological weight regimes than the last.

(defvar *act-ladder* '(relu gelu swish mish))

(defun activation-successor (act)
  (let ((pos (position act *act-ladder*)))
    (if (and pos (< pos (1- (length *act-ladder*))))
        (nth (1+ pos) *act-ladder*)
        'mish)))

;; ── Growth policy ─────────────────────────────────────────────────

(defun should-grow-p (current-loss prev-loss epoch)
  (and (> epoch 10)
       (> current-loss 0.15)
       (< (abs (- current-loss prev-loss)) 0.002)))

(defun growth-width (net)
  (min 512 (* 2 (width-of net :last-hidden))))

;; ── Training state ────────────────────────────────────────────────

(defvar *prev-loss*    1.0)
(defvar *growth-count* 0)
(defvar *max-growths*  5)

;; ── Training loop with live self-modification ─────────────────────

(with-training (net    mnist-net)
               (data   *train* :batch 64 :shuffle t)
               (loss   cross-entropy)
               (opt    (schedule (adam 0.001) :warmup 200 :decay cosine))
               (epochs 200)

  ;; 1. Vanishing gradient repair — every 5 epochs
  (when (zero-mod epoch 5)
    (let ((grad-norms (layer-gradient-norms mnist-net)))
      (loop for idx from 0
            for norm in grad-norms
            when (< norm 1e-5)
            do (let* ((cur  (layer-activation mnist-net idx))
                       (next (activation-successor cur)))
                 (format t "~&Epoch ~d: Layer ~d dying (~a → ~a)~%"
                         epoch idx cur next)
                 (rewrite mnist-net :layer idx :pos 2 next)))))

  ;; 2. Depth growth under stagnation
  (when (and (< *growth-count* *max-growths*)
             (should-grow-p current-loss *prev-loss* epoch))
    (let ((w (growth-width mnist-net)))
      (format t "~&Epoch ~d: Growing → width ~d~%" epoch w)
      (rewrite mnist-net :insert-before :output
        (dense w relu :drop 0.15))
      (incf *growth-count*)))

  ;; 3. Dropout annealing — every 25 epochs
  (when (zero-mod epoch 25)
    (let ((new-drop (max 0.05
                         (* 0.85 (dropout-of mnist-net :layer 1)))))
      (rewrite mnist-net :layer 1 :key :drop new-drop)))

  ;; 4. Logging — every 10 epochs
  (when (zero-mod epoch 10)
    (format t "~&Epoch ~d | Loss: ~,4f | Val: ~,2f% | Layers: ~d~%"
            epoch current-loss
            (* 100 (evaluate mnist-net *val* :metric :accuracy))
            (layer-count mnist-net)))

  ;; 5. Early stopping at target accuracy
  (when (> (evaluate mnist-net *val* :metric :accuracy) 0.993)
    (format t "~&Target reached at epoch ~d.~%" epoch)
    (signal 'stop-training))

  (setf *prev-loss* current-loss))

;; ── Results ───────────────────────────────────────────────────────

(format t "~&~%Growth events: ~d~%Modification history:~%" *growth-count*)
(dolist (entry (history mnist-net))
  (format t "  ~s~%" entry))
(inspect mnist-net)

;; ── Save the evolved architecture back to disk ────────────────────
;; The evolved defnet form is written as readable Lisp source.
;; Next load gets the architecture that training discovered.

(with-open-file (out "mnist-evolved.lisp"
                     :direction :output :if-exists :supersede)
  (write (inspect mnist-net) :stream out))

(format t "~&Evolved architecture saved → mnist-evolved.lisp~%")

The network starts with two layers — a single hidden layer of width 64 and an output layer. Depending on the training dynamics, it may end anywhere from three layers to seven, with widths ranging from 64 to 512, activation functions upgraded from relu toward gelu or swish in any layer showing vanishing gradients, and dropout rates annealing from 0.15 toward 0.05. The final architecture is not specified by the programmer. It is discovered.

The final five lines are the most important. The evolved architecture is written back to disk as a defnet form — readable LISP source code. The next time anyone loads mnist-evolved.lisp, they get not a weights file but a program: the specific network that this training run discovered, expressed in the same language used to write the original two-layer starting point. The architecture and the code are the same thing. They have always been the same thing. This is what the paper has been arguing.

Formal Grammar of NeuralLISP Core Forms

The following EBNF sketch covers the seven primitive forms, their principal companion and derived forms, and the new defsymbolic and with-rewrite-sandbox forms introduced in this revision. It is not a complete grammar — a full specification would require defining the host Common Lisp grammar, the tensor type system, and the macro expansion rules — but it is sufficient to unambiguously parse any NeuralLISP program that uses only the forms specified in this paper. Terminals are in monospace; non-terminals in italic; | denotes alternation; { } denotes zero or more; [ ] denotes optional.

;; ── TOP-LEVEL FORMS ──────────────────────────────────────────────

program       ::= { top-form }
top-form      ::= defactivation
               |  defweights
               |  defloss
               |  defnet
               |  defmodule
               |  defagent
               |  defsymbolic
               |  with-training
               |  with-rewrite-sandbox
               |  cl-form              ; any standard Common Lisp form

;; ── ACTIVATION DEFINITION ────────────────────────────────────────

defactivation ::= "(" "defactivation" symbol
                    ":forward"  lambda-form
                    ":backward" ( lambda-form | ":jacobian" )
                  ")"

activation    ::= symbol               ; must name a defactivation
               |  "(" "compose" { activation } ")"
               |  "(" "partial" activation { expr } ")"
               |  lambda-form          ; gradient-traced lambda

;; ── LAYER FORMS ──────────────────────────────────────────────────

layer-form    ::= dense-form
               |  conv-form
               |  recur-form
               |  attend-form
               |  output-form
               |  norm-form
               |  pool-form
               |  flatten-form

dense-form    ::= "(" "dense"  width activation
                    [ ":drop"    float ]
                    [ ":init"    init-strategy ]
                    [ ":weights" symbol ]
                    [ ":as"      symbol ]
                  ")"

conv-form     ::= "(" "conv"   filters activation kernel
                    [ ":stride"  shape-2d ]
                    [ ":pad"     shape-2d ]
                    [ ":drop"    float ]
                    [ ":as"      symbol ]
                  ")"

recur-form    ::= "(" "recur"  width cell-type
                    [ ":bidir"   bool ]
                    [ ":drop"    float ]
                    [ ":layers"  integer ]
                    [ ":as"      symbol ]
                  ")"

attend-form   ::= "(" "attend" heads dim
                    [ ":drop"    float ]
                    [ ":mask"    mask-type ]
                    [ ":as"      symbol ]
                  ")"

output-form   ::= "(" "output" width activation ")"

norm-form     ::= "(" "norm"   width
                    [ ":type"    ( ":layer" | ":batch" | ":rms" ) ]
                  ")"

pool-form     ::= "(" "pool"   shape-2d pool-strategy ")"

flatten-form  ::= "(" "flatten" ")"

;; ── NETWORK DEFINITION ───────────────────────────────────────────

defnet        ::= "(" "defnet" symbol "(" { symbol } ")"
                    net-body
                  ")"

net-body      ::= thread-form
               |  let-form             ; let / let* for skip connections
               |  values-form          ; multiple return values
               |  cl-expr              ; any expression returning a tensor

thread-form   ::= "(" "->" expr { thread-stage } ")"

thread-stage  ::= layer-form
               |  symbol               ; named defnet or defmodule
               |  "(" symbol { expr } ")"  ; function call

;; ── MODULE DEFINITION ────────────────────────────────────────────

defmodule     ::= "(" "defmodule" symbol "(" symbol { symbol } ")"
                    net-body
                  ")"
; First argument is the input tensor; remaining are dimension parameters.
; Weight tensors allocated fresh per call via with-fresh-parameters.

;; ── AGENT DEFINITION ─────────────────────────────────────────────

defagent      ::= "(" "defagent" symbol
                    ":obs-dim"    integer
                    ":act-dim"    integer
                    [ ":hidden"    "(" { integer } ")" ]
                    [ ":gamma"     float ]
                    [ ":gae-lambda" float ]
                    [ ":entropy-coeff" float ]
                    [ ":value-coeff"   float ]
                    [ ":max-grad-norm" float ]
                    [ ":buffer-size"   integer ]
                  ")"
; Expands to: policy defnet (obs→hidden→act softmax),
;             value  defnet (obs→hidden→1 linear),
;             nl-agent struct with replay buffer.

agent-act     ::= "(" "agent-act" symbol tensor
                    [ ":greedy" bool ]
                  ")"

agent-train   ::= "(" "agent-train" symbol
                    ":states"  tensor
                    ":actions" tensor
                    ":rewards" tensor
                    ":next-states" tensor
                    ":dones"   tensor
                  ")"

with-episodes ::= "(" "with-episodes"
                    "(" "agent" symbol ")"
                    "(" "env"   expr   ")"
                    "(" "episodes" integer ")"
                    { cl-form }
                  ")"

;; ── SYMBOLIC RULES (NEW IN V5) ───────────────────────────────────

defsymbolic   ::= "(" "defsymbolic" symbol mode
                    [ ":predicate" lambda-form ]
                    [ ":penalty"   lambda-form ]  ; soft rules only
                    [ ":weight"    float ]         ; soft rules only
                    [ ":invariant" lambda-form ]   ; hard rules only
                    [ ":message"   string ]
                    [ ":category"  keyword ]
                  ")"

mode          ::= ":soft" | ":hard"

;; ── REWRITE SANDBOX (NEW IN V5) ──────────────────────────────────

with-rewrite-sandbox ::= "(" "with-rewrite-sandbox"
                            "(" "net" symbol ")"
                            { cl-form }
                            [ "(" ":validate" expr ")" ]
                            [ "(" ":approve" approval-mode ")" ]
                            [ "(" ":rate-limit" integer ":per" interval ")" ]
                          ")"

approval-mode ::= ":human" | ":auto" | lambda-form
interval      ::= ":epoch" | ":episode" | ":second"

;; ── TRAINING ─────────────────────────────────────────────────────

with-training ::= "(" "with-training"
                    "(" "net"    symbol [ ":device" device ] ")"
                    "(" "data"   expr   [ ":batch" integer ]
                                       [ ":shuffle" bool ]
                                       [ ":devices" device-list ] ")"
                    "(" "loss"   loss-expr ")"
                    "(" "opt"    optimizer-expr ")"
                    "(" "epochs" integer ")"
                    [ "(" ":mode" training-mode ")" ]
                    { cl-form }        ; body — evaluated each epoch
                  ")"

;; ── TERMINALS ────────────────────────────────────────────────────

width         ::= integer
filters       ::= integer
heads         ::= integer
dim           ::= integer              ; must be divisible by heads
kernel        ::= shape-2d
shape-2d      ::= "(" integer integer ")"
cell-type     ::= "lstm" | "gru" | "rnn"
pool-strategy ::= "max" | "avg" | "global"
mask-type     ::= ":causal" | ":padding" | nil
init-strategy ::= ":xavier" | ":he" | ":random" | ":zero" | ":frozen"
training-mode ::= ":supervised" | ":adversarial" | ":actor-critic"
device        ::= ":cpu" | ":cuda" | "(" ":cuda" integer ")"
bool          ::= t | nil

The grammar is intentionally minimal. Any expression producing a tensor is a valid cl-expr in net-body and a valid thread-stage — this is the mechanism by which residual connections (+), skip connections (cat), ensemble operations (mean), and conditional dispatch (if) appear in network bodies without requiring special syntax. The uniformity of the tensor type is what makes the grammar this concise.
